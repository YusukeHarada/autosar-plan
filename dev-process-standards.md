## 1. 自動車業界の開発プロセス標準の全体像

### 1.1 主要標準の関係と使い分け

自動車業界には複数の「標準」が存在するが、それぞれ扱う対象が異なる。混同されやすいため、最初に整理する。

| 標準 | 種別 | 対象 | 一言で言うと |
|------|------|------|--------------|
| **ASPICE**（Automotive SPICE） | プロセス標準 | ソフトウェア開発プロセスの品質 | 「どのように作ったか」を評価する |
| **ISO 26262** | 機能安全規格 | 電子電気システムの安全設計 | 「危険に対してどう設計したか」を証明する |
| **ISO/SAE 21434** | サイバーセキュリティ規格 | 車両のサイバーセキュリティ | 「サイバー攻撃にどう対処したか」を管理する |
| **SAFe**（Scaled Agile Framework） | 開発手法・フレームワーク | 大規模Agile開発の組織運営 | 「複数チームでどう開発を回すか」の方法論 |
| **Agile / Scrum** | 開発手法 | チーム単位のイテレーション開発 | 「短いサイクルで動くものを作り続ける」手法 |

**重要な区別：**

- **プロセス標準（ASPICE等）** は「開発の進め方・管理の仕方」を定める。成果物の安全性を直接保証するものではない。
- **機能安全規格（ISO 26262等）** は「製品が安全であること」を設計・検証で証明するための技術的要求事項を定める。
- この2つは独立しており、ASPICE準拠だからといって機能安全が確保されているわけではない。ただし、**ISO 26262はASPICEと整合するプロセスを前提としており、両者を並行して適用するのが自動車業界のスタンダードである。**

### 1.2 なぜ建設機械でも自動車業界の標準が参照されるのか

建設機械は農業機械・建設機械向けの機能安全規格（ISO 25119）を持っているが、ソフトウェア開発プロセスに特化した業界固有の標準は事実上存在しない。

その結果、以下の理由から自動車業界の標準が参照されている。

1. **Classic AUTOSARの採用：** Classic AUTOSARはもともと自動車のECU（Electronic Control Unit）向けに策定されたアーキテクチャであり、その開発エコシステム・ツール・委託先企業はすべて自動車業界の標準（ASPICEを含む）を前提としている。
2. **Tier1・委託先の体制：** Classic AUTOSAR対応のBSP（Board Support Package）やBSW（Basic Software）を開発できる委託先の多くは、OEM（自動車メーカー）向けにASPICE対応の実績を持っており、同じプロセスを適用することで品質が担保しやすい。
3. **ISOとの整合性：** ISO 25119はISO 26262をベースとしており、ISO 26262と整合するASPICEを適用することで、農機・建機向け機能安全への対応が自然にしやすくなる。

## 2. ASPICE（Automotive SPICE）の詳細

### 2.1 ASPICEとは

**ASPICE（Automotive Software Process Improvement and Capability dEtermination）** は、ソフトウェア開発プロセスの能力を評価・改善するためのフレームワークである。2005年に欧州のOEM連合（AUDI・BMW・Daimler・Ford・GM・VW等）が策定し、現在はAutomotive SPICE PAMとして管理されている。

#### CMMIとの関係

- **CMMI（Capability Maturity Model Integration）** はCMU（カーネギーメロン大学）が開発した汎用的なプロセス能力モデルであり、ASPICEはこれを自動車ソフトウェア向けにカスタマイズしたものと考えると理解しやすい。
- ASPICEは **ISO/IEC 33000シリーズ**（プロセスアセスメントのISO標準）に準拠して設計されており、国際規格との整合性がある。
- ISO/IEC 12207（ソフトウェアライフサイクルプロセス）をベースにしており、「どのプロセスを実施すべきか」の定義はISO/IEC 12207に従っている。

### 2.2 プロセスアセスメントモデル（PAM）の構造

ASPICEのプロセスは以下のカテゴリに分類される。

#### ソフトウェアエンジニアリング（SWE）プロセス ← 最重要

| プロセス | 名称 | 概要 |
|----------|------|------|
| **SWE.1** | ソフトウェア要求分析 | システム要求をソフトウェア要求に落とし込む |
| **SWE.2** | ソフトウェアアーキテクチャ設計 | 全体構造・モジュール分割の設計 |
| **SWE.3** | ソフトウェア詳細設計・ユニット構築 | モジュール単位の詳細設計とコーディング |
| **SWE.4** | ソフトウェアユニット検証 | ユニットテスト（単体テスト） |
| **SWE.5** | ソフトウェア統合・統合テスト | モジュールを組み合わせた結合テスト |
| **SWE.6** | ソフトウェア適格性確認テスト | システム要求を満たすかの受入テスト |

#### システムエンジニアリング（SYS）プロセス

| プロセス | 名称 | 概要 |
|----------|------|------|
| **SYS.1** | 要求引き出し | ステークホルダーの要求を収集する |
| **SYS.2** | システム要求分析 | ステークホルダー要求からシステム要求を定義 |
| **SYS.3** | システムアーキテクチャ設計 | ハードウェア・ソフトウェアの分割を含む全体設計 |
| **SYS.4** | システム統合・統合テスト | H/W・S/Wを組み合わせたシステムテスト |
| **SYS.5** | システム適格性確認テスト | 顧客要求の充足確認 |

#### サポートプロセス（SUP）・管理プロセス（MAN）

| プロセス | 主な内容 |
|----------|----------|
| **SUP.1** | 品質保証 |
| **SUP.8** | 構成管理（バージョン管理・変更管理） |
| **SUP.9** | 問題解決管理（不具合トラッキング） |
| **SUP.10** | 変更要求管理 |
| **MAN.3** | プロジェクト管理 |
| **MAN.5** | リスク管理 |

### 2.3 Capability Level（CL）の定義

| CL | 名称 | 実態 |
|----|------|------|
| **CL0** | 不完全 | プロセスが実施されていないか、目標を達成できていない |
| **CL1** | 実施済み | プロセスが実施され、目標を達成しているが、管理は不十分 |
| **CL2** | 管理済み | プロセスが計画・監視・制御されており、成果物が管理されている |
| **CL3** | 確立済み | 標準化されたプロセスが組織として定義・適用されている |
| **CL4** | 予測可能 | プロセスが定量的に管理されている |
| **CL5** | 最適化 | プロセスの継続的改善が行われている |

**実際に求められるレベル：**

- OEM（自動車メーカー）がTier1（一次委託先）に要求するのは、一般的に **CL2〜CL3** である。
- CL2は「ちゃんと管理して作っている証拠がある」レベルであり、最初の目標として現実的。
- CL3以上はプロセスの組織標準化が必要であり、中規模以上の組織でないと維持が難しい。
- 建設機械の社内開発・小規模委託では、まず **CL1→CL2** の達成を目指すのが現実的な出発点である。

### 2.4 制御系ECU（Classic AUTOSAR）におけるASPICEの適用状況

#### 重視されるプロセスエリア

Classic AUTOSAR（RTOS系・制御系）では以下が特に重視される。

1. **SWE.1〜SWE.6**（ソフトウェア開発の上流〜下流の一貫性）
2. **SUP.8**（構成管理：ArXMLファイルのバージョン管理を含む）
3. **SUP.9 / SUP.10**（不具合・変更管理：制御ソフトの変更は安全への影響があるため厳格な管理が必要）
4. **SYS.2 / SYS.3**（システム要求・アーキテクチャ：BSWとASW（Application Software）の責任分担の明確化）

#### 証跡の典型的なセット

ASPICEアセスメントでは「証跡（Work Product）」と呼ばれるドキュメント・成果物の存在と品質が確認される。Classic AUTOSAR開発での典型的なセットを以下に示す。

| プロセス | 主な証跡 |
|----------|----------|
| SWE.1 | ソフトウェア要求仕様（SRS）、要求トレーサビリティマトリクス |
| SWE.2 | ソフトウェアアーキテクチャ設計書、インターフェース定義（ArXML含む） |
| SWE.3 | 詳細設計書（モジュール単位）、コーディング規約と遵守証跡 |
| SWE.4 | ユニットテスト仕様・結果、カバレッジレポート |
| SWE.5 | 統合テスト仕様・結果、HIL（Hardware in the Loop）テストレポート |
| SWE.6 | 受入テスト仕様・結果、トレーサビリティ（要求〜テスト） |
| SUP.8 | バージョン管理ログ、ベースライン管理記録 |
| SUP.9/10 | 不具合報告書（DR）、変更要求（CR）と承認記録 |

#### 外部委託時の確認事項

委託先のASPICEレベルを確認せずに発注することは、品質上のリスクになる。確認すべきポイントは以下のとおり。

- **ASPICEアセスメント結果の提示を求める：** 第三者機関によるアセスメント報告書が最も信頼性が高い。
- **対象プロセスとCLを確認：** 「ASPICE対応」とうたっていても、一部プロセスのみCL2で他はCL1以下という場合がある。
- **証跡のサンプルレビュー：** 実際の過去プロジェクトの証跡（SRS・設計書・テスト仕様）のサンプル提示を求める。
- **プロセスエンジニアの在籍確認：** 証跡作成・管理を専門に担当するプロセスエンジニアが在籍しているか確認する。

### 2.5 ASPICEアセスメントの費用感・工数感

第三者機関によるASPICEアセスメントの費用感は以下のとおり（あくまで参考値。要確定）。

| 規模・スコープ | 費用感（参考） | 期間 |
|---------------|--------------|------|
| 小規模プロジェクト（SWE中心、CL1〜2） | 数百万円〜 | 1〜2週間のアセスメント |
| 中規模（SWE+SYS、CL2〜3） | 500万〜1,000万円程度 | 2〜4週間のアセスメント |
| 組織全体のCL3認定 | 1,000万円以上 | 数ヶ月の改善活動＋アセスメント |

社内での自己アセスメント（Internal Assessment）から始めることで、外部アセスメントのコストを抑えることができる。まず内部でギャップ分析を行い、改善後に外部アセスメントを受けるのが現実的なアプローチである。

## 3. Linux系のプロセス標準

ここが現時点で最も答えが定まっていない領域である。正直に整理する。

### 3.1 ASPICEのLinux系への適用状況（現状）

#### ASPICEはLinux系に「適用できるか」という問いへの回答

**理論上は適用可能だが、実運用では多くの摩擦が生じる。**

ASPICEのプロセスフレームワーク自体は特定の技術に依存しておらず、Linux系のHMI・クラウド通信・ゲートウェイ開発にも適用は可能である。しかし、以下の点で古典的なウォーターフォール型開発を前提としたASPICE（特にVer.3.1以前）との齟齬が生じる。

| 課題 | 内容 |
|------|------|
| **リリースサイクルの違い** | 制御系ECUは年1〜2回のリリース。Linux系（HMI・クラウド）は月次〜週次のリリースが一般的。ASPICEの証跡作成が間に合わない。 |
| **Agileとの相性** | ASPICEは各工程（要求定義→設計→実装→テスト）の完了と証跡を前提とする。スプリント単位で動くAgile開発では工程の区切りが曖昧になる。 |
| **サービス指向アーキテクチャとの齟齬** | Adaptive AUTOSARやマイクロサービスでは「モジュール」より「サービス単位」で開発・テストが行われる。SWE.2の「アーキテクチャ設計」の解釈が困難になる。 |
| **OSSの利用** | Linuxカーネル・ミドルウェアはOSSを多用する。これらのSWE証跡（設計書・テスト結果）は存在しないため、ASPICE適用の境界をどこに引くか問題になる。 |

#### 実際に適用されているケース

Adaptive AUTOSAR対応を進める自動車OEMの一部では、ASPICEをAdaptive AUTOSAR開発にも適用しようとする動きがある。ただし、以下のような現実がある。

- **BIG OEM（VW・BMWグループ等）** は社内でAdaptive AUTOSARの開発プロセスを標準化しているが、そのプロセスをASPICEの文脈で整理しているというよりも、「Agile + CI/CD + ISO 26262対応」のフレームワークを独自に構築しているケースが多い。
- **Tier1サプライヤー（Bosch・Continental・ZF等）** はASPICEのプロセスをAgile対応に拡張した社内標準を持っており、それをAdaptive AUTOSAR開発にも適用している。ただし、その詳細は非公開であることがほとんど。
- **業界共通の答えはまだ出ていない** というのが2026年時点の正直な状況である。

### 3.2 Linux系で使われている代替・補完アプローチ

#### 3.2.1 ASPICE for Agile（Scrum対応PAM）

**ASPICE for Agile** は、ASPICE PAMをAgile開発（特にScrum）に対応させた拡張版であり、2019年にAutomotive SIG（自動車業界の業界団体）が公開した。

**概要：**

- SprintをASPICEのプロセスにマッピングする考え方を提供する。
- 「SWE.1（要求分析）→Sprint Backlog」「SWE.3（実装）→Sprint内の開発」のように対応付けを行う。
- Sprint Reviewの成果物をASPICEの証跡として活用する方法を示す。

**実態：**

- 公式ドキュメントはあるが、実際に現場で運用できているケースはまだ少ない。
- 「スプリント単位の証跡」が本当にASPICEアセスメントで受け入れられるかは、アセッサー（評価者）の解釈によって差がある。
- ツール（Jira等のアジャイル管理ツール）とASPICEの証跡管理ツールの連携が技術的なボトルネックになることが多い。

**評価：** 方向性は正しいが、現時点では「理念としては有効」であり「実運用の成熟度は低い」。

#### 3.2.2 ISO/IEC 33000シリーズとの関係

**ISO/IEC 33000シリーズ** はプロセスアセスメントの国際標準であり、ASPICEはこの標準に準拠している。

- Linux系の開発にASPICEを適用しない場合でも、ISO/IEC 33000の枠組みに基づいた独自のプロセスアセスメントモデルを定義することは理論上可能である。
- ただし、実際に独自モデルを定義・運用するには相当の工数が必要であり、小〜中規模の開発組織では現実的でない。

#### 3.2.3 SAFe（Scaled Agile Framework）の採用状況

**SAFe（Scaled Agile Framework）** は複数のAgileチームを組織全体で調整するためのフレームワークであり、プロセス標準ではなく「組織運営の方法論」である。

自動車業界での採用状況：

- VW・BMW・Stellantisなどの大手OEMの一部がSAFeを採用しているが、主に「OEMの社内開発組織」における適用であり、Tier1委託先との間でSAFeを適用するケースはまだ限定的。
- SAFe自体はASPICEを排除するものではなく、「SAFe + ASPICE for Agile」を組み合わせる試みもある。
- 建設機械の規模感（数十人〜数百人規模）では、SAFeの全体フレームワークより、ScaleなしのScrumまたはSAFeの一部（PI Planning等）を選択的に導入するアプローチが現実的。

#### 3.2.4 Automotive Agile / Agile@Scale の動向

- **VW（フォルクスワーゲン）グループ** はソフトウェア子会社「CARIAD」でSAFeベースのAgile開発を推進しているが、2024年に大規模な組織再編を行っており、プロセスの安定度は不明。
- **BMW** はBMW Group ITでAgile@ScaleとSAFeを組み合わせた独自フレームワークを運用している。
- **Stellantis** はAgile導入を進めているが、各ブランド間のプロセス統一が課題とされている。
- いずれも大規模組織特有の課題（組織の慣性・プロセス変更コスト）に直面しており、「うまくいっている」と言い切れる事例は少ない。

#### 3.2.5 Adaptive AUTOSAR固有のプロセス考慮点

Adaptive AUTOSARはClassic AUTOSARとアーキテクチャが根本的に異なるため、プロセスも異なる考え方が必要である。

| 側面 | Classic AUTOSAR | Adaptive AUTOSAR |
|------|----------------|-----------------|
| 開発単位 | モジュール（SWC: Software Component） | Adaptive Application / Service |
| 通信方式 | シグナル指向（CAN・J1939） | サービス指向（SOME/IP） |
| デプロイ | ROM書き込み（OTA以前は物理書き換え） | OTA（Over The Air）での動的更新 |
| プロセス上の課題 | SWCの要求・設計・テストの証跡管理 | サービス単位の独立デプロイ・バージョン管理 |

Adaptive AUTOSARでは「ara::com」（SOME/IPを抽象化した通信API）を使ったサービス間通信が中心となるため、SWE.2（アーキテクチャ設計）ではサービスインターフェース定義（SOME/IPサービスディスカバリを含む）の管理が証跡として必要になる。

### 3.3 クラウド・IoT系のプロセス標準

#### 使われているアプローチ

AWS・Azure等のクラウドIoT開発では、自動車業界的な「プロセス標準への準拠」という考え方よりも、以下のアプローチが主流である。

| アプローチ | 概要 |
|-----------|------|
| **DevSecOps** | 開発・セキュリティ・運用を継続的に統合する。CI/CDパイプラインにセキュリティテストを組み込む。 |
| **ISO/IEC 25010** | ソフトウェア品質モデル。機能適合性・信頼性・性能効率性・セキュリティ等の品質特性を定義。 |
| **AWS Well-Architected Framework** | AWSでのシステム設計のベストプラクティス（セキュリティ・信頼性・パフォーマンス・コスト等）。 |
| **NIST Cybersecurity Framework** | サイバーセキュリティリスク管理のフレームワーク。ISO/SAE 21434との整合性がある。 |

#### DevSecOpsとASPICEの関係

DevSecOpsとASPICEは「対立するもの」ではなく、**対象とする課題が異なる**。

- ASPICEは「要求→設計→実装→テストのプロセスが正しく管理されているか」を問う。
- DevSecOpsは「継続的なデリバリーとセキュリティの統合をどう実現するか」を問う。
- CI/CDパイプラインのログ・自動テスト結果をASPICEの証跡として活用する試みはあるが、標準的な方法論はまだ確立されていない。

**実務的な整理：** クラウド・IoT連携部分はASPICEの適用範囲外とし、代わりにISO/SAE 21434（サイバーセキュリティ）の要求事項に従ったTARA（脅威分析・リスクアセスメント）を実施することで、品質・セキュリティの担保を行うアプローチが現実的である。

#### OTAリリース管理のプロセス標準化

OTA（Over The Air）によるソフトウェア更新は、以下の観点からプロセス標準化が必要である。

- **変更管理（SUP.10相当）：** どのバージョンをどの車両・機体に配信したかの追跡管理。
- **ロールバック戦略：** 更新失敗時の切り戻し手順の定義と証跡。
- **機能安全への影響評価：** ISO 25119に基づく変更影響分析（OTAでも安全要求は変わらない）。
- **サイバーセキュリティ：** ISO/SAE 21434に基づくOTA更新の認証・完全性検証。

現時点でOTAに特化した業界共通のプロセス標準は存在しないが、UNECEのWP.29（車両法規の国際調和）が定める **UNECE WP.29 R156**（ソフトウェア更新管理システム）が参照されることが増えている。建設機械への直接適用義務はないが、設計思想の参考にはなる。

## 4. 現状の課題整理

### 4.1 制御系（Classic AUTOSAR / RTOS）の課題

#### ASPICEが「形式的適用」になっていることが多い理由

Classic AUTOSAR開発でASPICEを適用している委託先でも、証跡が形式的になっていることは珍しくない。主な原因は以下のとおり。

- **証跡作成が開発の後付けになる：** 実装が先行し、設計書・要求仕様があとから辻褄合わせで作られる。要求と実装の不一致が隠れる。
- **トレーサビリティが名目上：** 要求ID→設計→コード→テストのトレーサビリティマトリクスは存在するが、実際の変更が反映されておらず陳腐化している。
- **プロセスエンジニアが技術から孤立：** 証跡管理を担うプロセスエンジニアが開発の実態を知らず、開発者も証跡の意味を理解していない。
- **アセスメント対策としての運用：** アセスメントの直前に証跡を整備し、日常開発はASPICEと関係なく進んでいる。

#### 証跡作成がボトルネックになる典型パターン

1. スケジュール圧迫により「実装先・証跡後」になる
2. ArXMLファイルの変更と設計書の変更が同期されない（SUP.8の形骸化）
3. テスト仕様書が存在するがテスト実施の根拠記録がない
4. 不具合報告（DR）は存在するが、変更要求（CR）との連携がなく、変更の影響範囲評価がない

#### 外部委託先のASPICEレベルを確認せず発注している問題

「AUTOSAR対応実績あり」という自己申告だけで発注すると、以下のリスクが生じる。

- 委託先のASPICEレベルがCL1（または事実上CL0）であり、証跡が存在しない
- 証跡はあるが品質が低く、要求とコードのトレーサビリティが断絶している
- アセスメント経験がなく、発注元がアセスメントを求めた場合に対応できない

### 4.2 Linux系（HMI・クラウド・Adaptive AUTOSAR相当）の課題

#### 適用すべきプロセス標準が定まっていない

Linux系開発（HMI・ゲートウェイ・人検知・クラウド通信）に適用するプロセス標準は、現時点で社内で明確に決まっていない。これは当社固有の問題ではなく、**業界全体がまだ答えを出しきれていない領域**であるが、だからこそ「決めないまま放置する」という選択肢も避けるべきである。

#### Agileとウォーターフォール混在による摩擦

- 制御系はウォーターフォール（年次リリース・ASPICEプロセス）で開発している。
- Linux系はスプリント型で開発したい（HMIのUI改善・クラウド連携機能追加等）。
- 両者が連携する部分（例：HMIから制御系ECUへのコマンド送信仕様）の変更管理が曖昧になる。
- 「誰が何を決めて、どこで承認するか」のプロセスが定義されていないため、口頭確認が横行する。

#### 機能安全が必要なLinuxコンポーネントのプロセスが曖昧

人検知システム（カメラ・AI推論によるオペレータ周辺の人物検知）は、ISO 25119上の安全機能として位置付けられる可能性がある。この場合、Linuxベースのコンポーネントにも機能安全プロセスが必要になるが、以下の点が未整理である。

- ISO 25119上の安全度水準（ASIL相当）の割り当てが未確定
- Linuxが動作するSoCにおける安全機能の実現方式（QM部品の上で安全機能を実現するには追加の設計上の配慮が必要）
- 機能安全対応のLinuxミドルウェアをどこから調達するか

### 4.3 組織横断の課題

#### 制御系とLinux系で異なるプロセスを適用した場合の統合問題

将来的に制御系（Classic AUTOSAR / ASPICE）とLinux系（独自またはASPICE for Agile）で異なるプロセスを適用した場合、以下の統合問題が発生する。

- **変更管理の統合：** 制御系に影響するLinux側の変更をどのプロセスで管理するか
- **テスト結果のトレーサビリティ：** システムレベルテスト（SYS.4）の証跡に制御系・Linux系の両方を含める必要があるが、証跡フォーマットが異なると統合が困難
- **リリース管理の同期：** 制御系（低頻度リリース）とLinux系（高頻度リリース）のリリースサイクルを合わせる際のルールが未定義

#### 外部委託先との標準の不一致

- 委託先A（Classic AUTOSAR）：ASPICEで開発
- 委託先B（HMI開発）：独自のAgile開発（ASPICEなし）
- 自社：プロセス標準未定義

この状況では、システム統合時に品質基準・証跡フォーマット・変更管理手順がすべてバラバラになる。インターフェース仕様の合意や不具合対応の責任分担が口頭・メールベースになり、追跡不能になるリスクがある。

#### プロセス担当者（Process Engineer）不在問題

ASPICEを含む開発プロセス標準の適用・維持・改善を専任で担当するプロセスエンジニアが社内に存在しない場合、以下の問題が生じる。

- 委託先から提出される証跡の品質を評価できない
- 社内開発者へのプロセス教育が行われない
- プロセス改善のPDCAが回らない
- アセスメント時に社内の状況を説明・対応できる人材がいない

## 6. 建設機械への適用における現実解

### 6.1 「ASPICEを全部やる」が現実的でない理由

ASPICEのすべてのプロセスエリアをフルスコープで適用し、全プロセスでCL3を達成することは、大手Tier1サプライヤーであっても相当の投資が必要である。建設機械の自社開発組織（数十人規模）で同じことを目指すことは現実的でない。主な理由は以下のとおり。

- **工数コスト：** 証跡作成・管理の工数は、プロジェクト全体の15〜30%に達することがある（成熟度が低い段階では特に）。
- **人材：** プロセスエンジニア・アセッサーの育成・採用には時間とコストがかかる。
- **ROI（投資対効果）の観点：** 建設機械は自動車と異なり、リコール規制や型式認証でASPICEが法的に要求されているわけではない。過剰投資になる可能性がある。
- **委託先の現実：** AUTOSAR対応可能な委託先の中でも、建設機械向けに喜んで受注してくれる企業は限られており、ASPICE要件を厳しくすると候補が更に絞られる。

### 6.2 「小さく始める」ための優先プロセスエリア選定

以下の基準で優先順位を判断する。

**優先度：高（まず取り組む）**

| プロセス | 理由 |
|----------|------|
| **SWE.1**（ソフトウェア要求分析） | 要求が曖昧なまま開発が進む問題を根本解決する。他のすべての証跡の起点になる |
| **SUP.8**（構成管理） | ArXMLとソースコードのバージョン管理なしに品質は担保できない。GitやPLMツールで対応可能 |
| **SUP.9**（問題解決管理） | 不具合の記録・追跡はどの開発でも必須。JiraやRedmineで対応可能 |
| **SWE.6**（ソフトウェア適格性確認テスト） | 「完成の定義」を明確にする。テスト結果の記録が契約上の納品物になる |

**優先度：中（CL2達成後に取り組む）**

| プロセス | 理由 |
|----------|------|
| **SWE.2**（アーキテクチャ設計） | BSW・ASW・Linuxミドルウェアの責任境界を明確化する |
| **SWE.4・SWE.5**（ユニット・統合テスト） | テスト仕様と結果の管理。CI/CDと連携することで工数を最小化できる |
| **MAN.3**（プロジェクト管理） | プロジェクト計画・進捗管理の形式化 |

**優先度：低（将来検討）**

| プロセス | 理由 |
|----------|------|
| **SWE.3**（詳細設計・ユニット構築） | コーディング規約の適用等。まずはSWE.1〜2の充実が先 |
| **MAN.5**（リスク管理） | 形式的なリスク管理ではなく実効性が重要。成熟度が上がってから取り組む |
| **SUP.1**（品質保証） | プロセス全体の品質保証。他のプロセスが安定してから |

### 6.3 外部委託先選定でのASPICE要件の現実的な落としどころ

建設機械向けのClassic AUTOSAR開発委託先に対するASPICE要件の現実的な基準を以下に示す。

| 要件 | 必須 / 推奨 | 備考 |
|------|------------|------|
| SWE.1〜SWE.6のCL1以上 | **必須** | 最低限、各プロセスが実施されていることを確認 |
| SUP.8・SUP.9のCL1以上 | **必須** | 構成管理・問題管理がないと協業が困難 |
| SWE.2・SWE.6のCL2以上 | **推奨** | アーキテクチャと受入テストはCL2が望ましい |
| 第三者アセスメント受審実績 | **推奨** | 自己評価より信頼性が高い |
| プロセスエンジニアの在籍 | **推奨** | 証跡品質の継続的な維持に必要 |
| 組織全体のCL3達成 | 不要（当面） | 大手Tier1レベルの要件であり、現実的でない |

**実務的なアドバイス：**

- 契約前に「SWE.1・SWE.6・SUP.8の代表的な証跡（直近プロジェクトのもの）」の提示を必須条件とする。
- 提示された証跡が「形式的なものか実質的なものか」を評価できる社内の技術者を育成するか、コンサルタントを活用する。
- 完璧なASPICE適用を求めるよりも、「要求仕様→テスト結果のトレーサビリティが確認できること」を最低条件とし、段階的に要件を引き上げるアプローチが委託先との良好な関係を維持しやすい。
