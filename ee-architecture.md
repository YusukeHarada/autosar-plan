# E/Eアーキテクチャ変遷

建設機械向けプラットフォームソフトウェア移行を理解するための、自動車・建設機械E/E（電気電子）アーキテクチャの変遷と設計思想の解説。AUTOSARやSDV移行の「なぜ」を理解する基礎知識となる。

---

## 1. E/Eアーキテクチャとは

E/Eアーキテクチャ（Electrical/Electronic Architecture）とは、**車両・機械全体の電気・電子システムの構成設計**のこと。具体的には次の要素を含む。

- ECU（電子制御ユニット）の数・種類・配置
- ECU間の通信ネットワーク（CAN・LIN・Ethernet等）
- 電源供給の設計（電圧・冗長性）
- センサ・アクチュエータとのインタフェース
- ソフトウェアの分散方針

E/Eアーキテクチャは「どこに何の機能を置くか」を決める設計で、ハードウェア・ソフトウェア双方に大きく影響する。

---

## 2. アーキテクチャ変遷の全体像

```
1990年代          2000年代          2010年代          2020年代以降
分散型            ドメイン型         ゾーン型           集中型
（Distributed）  （Domain）        （Zonal）         （Centralized）
    │                │                 │                  │
ECUごとに         機能ドメインで       車体の物理ゾーンで    中央コンピュータに
独立した          ECUをまとめる        ECUをまとめる        機能を集約
通信ライン
```

---

## 3. 分散型アーキテクチャ（1990年代）

### 3.1 概要

最初期の車載電子システム。機能ごとに専用のECUを搭載し、センサ・アクチュエータと1対1で接続。

```
[エンジン制御] ── 独立ライン ── [エンジンセンサ群]
[ABS制御]     ── 独立ライン ── [ホイールセンサ]
[エアコン制御] ── 独立ライン ── [温度センサ]
```

### 3.2 特徴と限界

- **メリット**：シンプル・ECU間依存なし・故障影響が局所的
- **デメリット**：ECU数増加に伴いワイヤーハーネスが膨大化・重量増・コスト増。ECU間でデータ共有ができない

建設機械では現在もこの思想に近い「ユニットごとに完結した制御」が残っている場合が多い。

---

## 4. ドメイン型アーキテクチャ（2000年代〜2010年代）

### 4.1 概要

機能的に関連するECUを「ドメイン」としてまとめ、**ドメインコントローラ**（DCU: Domain Control Unit）がドメイン内を統括する。ドメイン間通信はバックボーンネットワーク（CAN・初期Ethernet）を介す。

```
                    ┌── パワートレインドメイン ──┐
                    │  [エンジン制御ECU]         │
バックボーン CAN ──── DCU  [AT制御ECU]           │
                    │  [アイドリング制御ECU]      │
                    └───────────────────────────┘

                    ┌── シャシードメイン ────────┐
バックボーン CAN ──── DCU  [ABS ECU]            │
                    │  [EPS ECU]               │
                    └───────────────────────────┘

                    ┌── ボディドメイン ──────────┐
バックボーン CAN ──── DCU  [ドア制御ECU]         │
                    │  [照明ECU]               │
                    └───────────────────────────┘
```

### 4.2 AUTOSARとドメイン型の関係

Classic AUTOSARは**ドメイン型アーキテクチャを前提として設計**されている。RTE（Runtime Environment）によってSWC（Software Component）を抽象化し、ECU間通信をPDU（Protocol Data Unit）で標準化することで、ドメイン内外のソフトウェア再利用性を高める。

### 4.3 ドメイン型の限界

- ECU数の増加が止まらない（現代の高級車では100〜150個以上のECUが搭載される車種も）
- ドメインをまたぐ機能（自動運転・ADAS等）の実現が難しい（複数DCUの協調が必要）
- ハーネス重量・コストがボトルネック
- OTAアップデートが複数のドメインにまたがる場合に管理が複雑

---

## 5. ゾーン型アーキテクチャ（2010年代後半〜）

### 5.1 概要

「機能」ではなく「車体の物理的な場所（ゾーン）」でECUをまとめる発想の転換。各ゾーンに**ゾーンコントローラ**を置き、センサ・アクチュエータとの物理的な配線を短縮する。ゾーンコントローラはバックボーンEthernet（TSN）で中央コンピュータと接続。

```
                    [車両中央コンピュータ（HPC）]
                           │ バックボーンEthernet（TSN）
          ┌────────────────┼────────────────┐
          │                │                │
  [前方ゾーンコントローラ] [後方ゾーンコントローラ] [コックピットゾーンコントローラ]
          │                │                │
  [前方センサ群]      [後方センサ群]      [ディスプレイ群]
  [前方アクチュエータ]  [後方アクチュエータ]
```

### 5.2 ゾーン型の利点

- ハーネス長の短縮（各ゾーンで物理的に近いデバイスをまとめる）
- センサ・アクチュエータの追加がゾーンコントローラの追加で対応可能
- 機能の上位コンピュータへの集約が容易になる

### 5.3 建設機械への類比

建設機械でゾーン型に類するアプローチを取るとすれば：

```
[アームゾーン]: 関節センサ・シリンダ圧センサ・電磁弁 → ゾーンコントローラ
[走行ゾーン]:  走行モータ・速度センサ・旋回センサ → ゾーンコントローラ
[キャビンゾーン]: 操作レバー・ディスプレイ・空調 → ゾーンコントローラ
```

ただし建設機械はECU数が自動車より少なく、現時点でゾーン型に移行するメリットは限定的。むしろドメイン型（Classic AUTOSAR）に整理してから将来を考える方が現実的。

---

## 6. 集中型アーキテクチャ（2020年代以降）

### 6.1 概要

演算能力の高いSoC（System on Chip）を搭載した**中央コンピュータ（HPC: High Performance Computer / VCU: Vehicle Computer Unit）**に機能を集約する。ソフトウェアはHPC上でVM（仮想マシン）やコンテナとして動作し、物理的なECUの数を大幅に削減する。

```
          ┌────────────────────────────────────┐
          │  中央コンピュータ（HPC）              │
          │  ┌──────────┐  ┌──────────────┐   │
          │  │ ADAS VM  │  │ ボディ制御VM  │   │
          │  └──────────┘  └──────────────┘   │
          │  ┌──────────┐  ┌──────────────┐   │
          │  │ IVI VM   │  │ AUTOSAR VM   │   │
          │  └──────────┘  └──────────────┘   │
          └────────────────────────────────────┘
                    │ Ethernet TSN
          ┌─────────┼─────────┐
  [ゾーンコントローラ] [センサハブ] [アクチュエータ]
```

### 6.2 Adaptive AUTOSARとの関係

集中型アーキテクチャでは、HPCのLinux/QNX上でAdaptive AUTOSARが動作する形が自動車業界の方向性。Adaptive AUTOSARの主要コンセプトである**SOA（Service-Oriented Architecture）**により、機能をサービスとして動的に配置・更新できる。

### 6.3 集中型の課題

- SoC・HPCのコストが高い
- Hypervisor上での機能安全認証（ISO 26262 / ISO 25119）の実績が少ない
- サイバーセキュリティ上の攻撃面が集中する（単一障害点リスク）
- 開発・検証の複雑性が増す

---

## 7. 各アーキテクチャの比較まとめ

| 項目 | 分散型 | ドメイン型 | ゾーン型 | 集中型 |
|---|---|---|---|---|
| ECU数 | 多い | 中程度 | 少ない（ゾーンコントローラ） | 少ない（HPC＋ゾーン） |
| ハーネス長 | 長い | 中程度 | 短い | 最短 |
| 機能の柔軟性 | 低 | 中 | 高 | 最高 |
| OTA対応 | 困難 | 部分対応 | 対応しやすい | 最適 |
| 機能安全対応 | 確立済み | 確立済み | 知見蓄積中 | 課題あり |
| 導入コスト | 低 | 中 | 高 | 最高 |
| 建設機械での現状 | 主流 | 移行対象 | 将来検討 | 長期検討 |

---

## 8. 建設機械の現状と移行方針

### 8.1 現状（2026年時点）

建設機械の多くは「分散型に近いドメイン型」の状態にある。ユニット（油圧・走行・アーム等）ごとに独立したECUが存在し、J1939 CANで接続されているが、アーキテクチャとして体系的に設計されていない場合がほとんど。

### 8.2 移行方針

```
現状（分散型に近い）          〜2029年                  2030年代
─────────────────────────────────────────────────────────────────
内製ECU群（設計なし）  →  Classic AUTOSAR    →  ゾーン型の要素を取り込みながら
J1939 CAN             →  ドメイン型整理      →  集中型へ段階移行
                      →  Ethernet導入
```

ドメイン型への整理（Classic AUTOSAR準拠）を「現在の正解」として、ゾーン型・集中型は将来技術と位置づける。

### 8.3 E/Eアーキテクチャ変遷がAUTOSAR移行に与える示唆

- Classic AUTOSARはドメイン型アーキテクチャの標準化として有効。今から導入しても10年以上は有用
- Adaptive AUTOSARはゾーン型・集中型のHPCに向けた規格。2028年以降の次世代機で検討
- E/Eアーキテクチャの変化はハードウェアと不可分。ソフトウェア移行計画はハード設計と並行して検討が必要
