# Bootloader / フラッシュプログラミング

建設機械向けECUのBootloader設計・UDS書き込みシーケンス・OTA連携の技術解説。OTA実装や委託先との仕様協議に必要な基礎知識をまとめる。

---

## 1. Bootloaderとは

Bootloader（ブートローダー）は、ECUの電源投入時に最初に起動する小さなプログラムで、**アプリケーションプログラムのフラッシュ書き込み（プログラミング）と起動管理**を担う。

```
電源ON
  │
  ▼
┌────────────────────────────────────┐
│ Bootloader（ROM/Flashの固定領域）   │
│  1. ハードウェア初期化              │
│  2. 書き込みリクエストの有無確認     │
│  3a. リクエストあり → フラッシュ書き込みモードへ
│  3b. リクエストなし → アプリケーション起動
└────────────────────────────────────┘
  │
  ▼
┌────────────────────────────────────┐
│ アプリケーション（制御ソフトウェア） │
└────────────────────────────────────┘
```

### 1.1 Bootloaderが必要な理由

- マイコンのフラッシュメモリへの書き込みは実行中のプログラム自身では行えない（書き込み対象の領域を実行しながら書き換えることができないため）
- 工場出荷後のファームウェア更新（バグ修正・機能追加）にはBootloaderが不可欠
- OTA（Over-The-Air）更新の実現にも必須

## 2. フラッシュメモリの構造

### 2.1 典型的なメモリマップ

```
高アドレス
┌──────────────────────────────┐
│  アプリケーションソフトウェア  │ ← Bootloaderが書き換える対象
│  （キャリブレーションデータ含む）│
├──────────────────────────────┤
│  Bootloader                  │ ← 原則として書き換えない（書き換えると起動不能になるリスク）
├──────────────────────────────┤
│  NVM / EEPROM エミュレーション │ ← 設定値・診断コード保存
低アドレス
└──────────────────────────────┘
```

### 2.2 デュアルバンク構成

大容量フラッシュでは「デュアルバンク（A/Bバンク）」構成が採用される。片方のバンクで稼働しながら、もう片方のバンクに新しいファームウェアを書き込む。書き込み完了後に起動バンクを切り替える方式。

```
バンクA（現行バージョン動作中）│バンクB（新バージョン書き込み中）
        ↓ 書き込み完了・検証OK
バンクA（待機）               │バンクB（新バージョン起動）
```

OTA更新で電源断が発生しても書き込み中のバンクが壊れるだけで、元のバンクから起動できる（ロールバック可能）。

## 3. UDSによるフラッシュプログラミング

### 3.1 UDS（Unified Diagnostic Services）とは

UDS（ISO 14229）は診断通信の国際規格で、**フラッシュプログラミングもUDSサービスとして定義**されている。工場での書き込みから、サービスショップでの更新、OTAまで同一プロトコルを利用できる。

### 3.2 プログラミングシーケンス

標準的なUDSプログラミングシーケンスは以下の通り。

```
テスター（書き込みツール / OTAゲートウェイ）       ECU（Bootloader）
        │                                              │
        │ ① DiagnosticSessionControl (0x10)            │
        │   → ExtendedDiagnosticSession or             │
        │     ProgrammingSession (0x02)                 │
        │──────────────────────────────────────────────▶│
        │                                              │ ECUはBootloaderへ遷移
        │ ② SecurityAccess (0x27)                      │
        │   → Seed要求 → Key計算 → アクセス許可         │
        │──────────────────────────────────────────────▶│
        │                                              │
        │ ③ RoutineControl (0x31)                      │
        │   → EraseMemory（フラッシュ消去）              │
        │──────────────────────────────────────────────▶│
        │                                              │
        │ ④ RequestDownload (0x34)                     │
        │   → データサイズ・アドレス通知                  │
        │──────────────────────────────────────────────▶│
        │                                              │
        │ ⑤ TransferData (0x36) × N回                  │
        │   → 分割データ転送（ブロックサイズ単位）         │
        │──────────────────────────────────────────────▶│
        │                                              │
        │ ⑥ RequestTransferExit (0x37)                 │
        │──────────────────────────────────────────────▶│
        │                                              │
        │ ⑦ RoutineControl (0x31)                      │
        │   → CheckProgrammingIntegrity（整合性確認）    │
        │──────────────────────────────────────────────▶│
        │                                              │
        │ ⑧ ECUReset (0x11)                            │
        │   → アプリケーション再起動                     │
        │──────────────────────────────────────────────▶│
```

### 3.3 SecurityAccess（セキュリティアクセス）

不正書き換えを防ぐためのチャレンジ・レスポンス認証。

```
テスター              ECU
  │                   │
  │ Seed要求 ────────▶│
  │                   │ 乱数（Seed）生成
  │ ◀──────── Seed ──│
  │                   │
  │ Key計算（Seed + シークレットキー → Hash等）
  │ Key送信 ──────────▶│
  │                   │ KeyをECU側でも計算・比較
  │ ◀──── 認証OK ────│
```

シークレットキーは委託先・ECUサプライヤ間で厳密に管理する必要がある。AUTOSARのSecOC（Secure Onboard Communication）とは別の仕組みだが、組み合わせて考える。

## 4. AUTOSAR Bootloader関連モジュール

### 4.1 Classic AUTOSARのBSWモジュール

| モジュール | 役割 |
|---|---|
| DCM（Diagnostic Communication Manager） | UDSサービスの受信・ルーティング。プログラミングセッション管理 |
| FLS（Flash Driver） | フラッシュメモリの消去・書き込みAPIを提供 |
| FEE（Flash EEPROM Emulation） | フラッシュをEEPROM的に使うためのドライバ |
| Fota（Firmware Over-The-Air） | OTA専用モジュール（AUTOSAR R20-11以降、要確定） |
| BootManager | アプリケーションとBootloaderの起動制御 |

### 4.2 分離BootloaderとAUTOSARの関係

Bootloader自体はAUTOSARの規格範囲外（ECUサプライヤの独自実装が多い）。AUTOSARアプリケーション側はDCMとFLSを使ってプログラミングセッションに対応する。

実装パターン：
1. **Bootloader独立型**：Bootloaderは独立したプログラムとしてECUに焼かれており、AUTOSAR BSWとは別に管理（一般的）
2. **AUTOSAR統合型**：AUTOSAR BSW上でBootloader機能も管理（高度な構成で要スキル）

## 5. OTA連携アーキテクチャ

### 5.1 OTA書き込みのデータフロー

```
クラウド（OTAサーバー）
    │ HTTPS / MQTT
    │ ファームウェアパッケージ配信
    ▼
Linuxゲートウェイ ECU
    │ ①パッケージ受信・署名検証
    │ ②バイナリ展開・対象ECU振り分け
    │ CAN FD / Ethernet
    ▼
対象 ECU（Classic AUTOSAR）
    │ ③UDSプログラミングシーケンス実行
    │ ④フラッシュ書き込み
    │ ⑤整合性確認
    ▼
更新完了・ECU再起動
    │ ⑥結果をゲートウェイ→クラウドへ報告
```

### 5.2 ファームウェアパッケージの形式

AUTOSARではUDS書き込みに用いるファームウェアイメージの形式として、**AUTOSAR Firmware Package（AFP）**や**SWE（Software Element）**の概念がある。実際には各ECUサプライヤが独自フォーマットを持つ場合も多いため、委託先との仕様整合が必要。

### 5.3 WP.29 R156との関連

国連の自動車規制（WP.29 Regulation No.156）はOTA書き込みのプロセス全体（セキュリティ・整合性確認・ロールバック・通知）を要件として定めている。建設機械は現時点でR156の直接適用対象外だが、OTAを実装する際は同等の考え方を参考にすることを推奨。

## 6. セキュリティ考慮事項

### 6.1 不正書き換えへの対策

| リスク | 対策 |
|---|---|
| 不正なファームウェアの書き込み | 署名検証（RSA / ECDSA等）。公開鍵をECU内に保持 |
| SecurityAccessの解析による認証突破 | シークレットキーのセキュアストレージ格納・HSM（Hardware Security Module）活用 |
| 通信経路での改ざん | TLS / MACによる通信保護（ゲートウェイ〜クラウド間） |
| バージョンダウングレード攻撃 | バージョンカウンタ管理・古いバージョンへの書き戻し禁止 |

### 6.2 HSM（Hardware Security Module）の役割

近年のマイコン（RH850・TC3xx等）にはHSMが内蔵されており、以下を実現する。

- 秘密鍵の安全な保管（HSM外部に鍵を出さない）
- ファームウェア署名の検証処理
- SecOC用のMACキー管理

AUTOSAR移行時にHSM対応を計画に含めることを推奨。

## 7. 建設機械固有の考慮事項

### 7.1 現場環境での書き込み

建設機械は工場外（現場・サービスショップ）でのファームウェア更新が発生するため：

- 専用の診断ツール（テスター）のUDS対応が必要
- 通信インタフェース：DoCAN（CAN上のISO 15765）が現実的
- 書き込み時間：CAN FDへの移行で高速化（CAN:〜数十分 → CAN FD:〜数分、要確定）

### 7.2 車両稼働中の制約

建設機械は現場で稼働中のため「エンジン停止・降車・書き込み」というシーケンスの徹底が必要。OTA書き込みは安全に停止できる状況での実施を必須条件とすること。

### 7.3 ECU数と更新順序

複数のECUが連携して動作する場合、更新順序の定義が必要（依存関係のあるECUを逆順で更新するとシステムが不整合になるリスク）。委託先との仕様書にECU間の依存関係と推奨更新順序を明記する。
