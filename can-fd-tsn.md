# CAN FD / Ethernet TSN 技術概念

建設機械向けプラットフォームソフトウェア移行における車載ネットワーク技術の解説。ハード選定・委託先との仕様協議に必要な技術基礎知識をまとめる。

---

## 1. 車載ネットワークの変遷と現状

### 1.1 なぜ通信規格が進化するのか

建設機械・自動車ともに、ECU数の増加と処理するデータ量の爆発的増加が技術進化の根本的な理由である。

```
1990年代: CAN (1Mbps)  → エンジン・トランスミッション制御
2000年代: CAN (250/500Kbps) → ドメイン内統合
2010年代: CAN FD (最大8Mbps) → ドメイン間帯域不足の解消
2020年代: Ethernet (100Mbps〜1Gbps+TSN) → 集中型アーキテクチャ・OTA・カメラ映像
```

建設機械では現状CAN（J1939）が主流だが、HMI高度化・クラウド連携・人検知カメラ統合に向けてEthernetへの段階移行が必要になっている。

---

## 2. CAN FD（CAN with Flexible Data-Rate）

### 2.1 概要

CAN FD（ISO 11898-1:2015）は従来CANの後継として2012年にBoschが策定した通信規格。「Flexible Data-Rate」の名の通り、データフェーズのビットレートを動的に変えられる。

### 2.2 従来CANとの主な違い

| 項目 | 従来CAN | CAN FD |
|---|---|---|
| 最大データ長 | 8バイト | 64バイト |
| アービトレーション速度 | 最大1Mbps | 最大1Mbps（同じ） |
| データフェーズ速度 | アービトレーション速度と同じ | 最大8Mbps（要確定） |
| フレームフォーマット | クラシックフレームのみ | クラシックCAN互換フレーム + FDフレーム |
| CRC | 15bit | 17bit / 21bit（データ長に応じて） |
| エラー検出率 | 高 | さらに向上 |

**重要**：CAN FDコントローラはクラシックCANフレームも送受信できる（後方互換性あり）。ただし、ネットワーク上にクラシックCANのみ対応のノードが1台でも存在すると、そのネットワーク全体でFDフレームは使用不可。

### 2.3 ビットレートの考え方（デュアルビットレート）

```
┌──────────────────────────────────────────────────────┐
│ CANフレームの構造                                       │
│                                                       │
│ [SOF][アービトレーション][制御][データ][CRC][ACK][EOF]  │
│  ↑                  ↑         ↑                       │
│  アービトレーション速度         データフェーズ速度         │
│  （最大1Mbps）                 （最大8Mbps）             │
└──────────────────────────────────────────────────────┘
```

- **アービトレーションフェーズ**：全ノードが同じ速度（バス上の衝突検出が必要なため低速）
- **データフェーズ**：アービトレーション後、送信ノードが1台に絞られるため高速化可能

実用的な組み合わせ例（要確定）：
- アービトレーション500Kbps + データ2Mbps
- アービトレーション500Kbps + データ5Mbps

### 2.4 建設機械への適用検討

**適用が有効な箇所**：
- ECU間の制御データ（油圧・電動アクチュエータ指令）：データ量増加への対応
- BootloaderによるECUファームウェア更新：8バイト制限が書き込みの足かせになっているケースで高速化
- J1939との共存：同一物理バスでJ1939（クラシックCAN）とCAN FDを混在させることは困難。ネットワーク設計で分離が基本

**制約**：
- バスの最大長が短くなる傾向（高速ほど終端処理・ケーブル品質に敏感）
- ハーネス・コネクタも対応品を選定する必要あり（委託先・ハードウェアベンダーと協議要）

### 2.5 J1939との関係

J1939はSAE（米自動車技術者協会）が策定した建設・農業・商用車向けのCAN上位プロトコル。CAN FDはJ1939の**物理層・データリンク層**を拡張するものであり、J1939のアプリケーション層（PGN・SPN定義）はCAN FD上でも利用可能。

ただし、SAE J1939の正式なCAN FD対応仕様（J1939-17）は策定済みだが、建設機械業界での採用事例はまだ少ないため、委託先との仕様協議で「どのレイヤーをどう対応させるか」を明確にする必要がある。

## 3. Ethernet TSN（Time-Sensitive Networking）

### 3.1 TSNとは何か

TSN（Time-Sensitive Networking）は、IEEE 802.1ワーキンググループが策定した**Ethernetに時刻同期・帯域保証・低遅延を付与する拡張規格群の総称**。

従来のEthernetはベストエフォート通信（遅延保証なし）であったが、TSNにより制御系通信に要求される確定的遅延（Deterministic Latency）を実現できる。

### 3.2 主要なTSN規格

| 規格 | 名称 | 役割 |
|---|---|---|
| IEEE 802.1AS | gPTP（Generalized PTP） | ネットワーク全体の時刻同期（±1μs精度、要確定） |
| IEEE 802.1Qbv | Time-Aware Shaper (TAS) | 時間スロット制御による優先フレームの確定的送信 |
| IEEE 802.1Qbu / 802.3br | Frame Preemption | 優先度の高いフレームが低優先フレームの送信を割り込み中断 |
| IEEE 802.1Qav | Credit-Based Shaper (CBS) | 音声・映像等の帯域平滑化 |
| IEEE 802.1Qcc | Centralized/Distributed Configuration | TSN設定の集中管理 |
| IEEE 802.1CB | Frame Replication and Elimination | 冗長経路による信頼性向上 |

### 3.3 TSNの動作原理（TASを例に）

```
時間軸
│←──── GCL（Gate Control List）サイクル ────→│
│                                              │
│ [制御フレーム優先]│[通常通信]│[制御フレーム優先]│
│ ゲートOPEN       │ゲートCLOSE│ ゲートOPEN      │
│                  │           │                 │
```

- GCL（Gate Control List）でポートごとに「どの時間帯にどのトラフィッククラスを通すか」をスケジューリング
- IEEE 802.1ASで全スイッチ・ECUの時刻を同期させることでスケジュール通りの動作が保証される

### 3.4 車載Ethernetの物理層規格

| 規格 | 速度 | 特徴 | 用途 |
|---|---|---|---|
| 100BASE-T1 (OPEN Alliance / IEEE 802.3bw) | 100Mbps | 非シールド単芯ツイストペア・軽量 | ECU間通信、センサ |
| 1000BASE-T1 (IEEE 802.3bp) | 1Gbps | 同上・高速 | ゲートウェイ、カメラ |
| 10BASE-T1S (IEEE 802.3cg) | 10Mbps | マルチドロップ対応（CANに近い構成） | CAN代替用途 |
| 10GBASE-T1 | 10Gbps | 次世代（現状研究段階） | 将来のバックボーン |

建設機械への適用は100BASE-T1 / 1000BASE-T1が現実的。標準的なRJ45 Ethernetではなく車載専用の単芯ケーブルであるため、ハーネス設計の見直しが必要。

### 3.5 SOME/IP との関係

SOME/IP（Scalable service-Oriented MiddlewarE over IP）はAUTOSARが定義するEthernet上のミドルウェアプロトコル。TSNはL2（データリンク層）の帯域・遅延保証、SOME/IPはL7（アプリケーション層）のサービス指向通信を担う。

```
アプリケーション層: SOME/IP（サービス定義・シリアライズ）
         ↑
トランスポート層:  UDP / TCP
         ↑
ネットワーク層:   IP
         ↑
データリンク層:   Ethernet + TSN（帯域・遅延保証）
         ↑
物理層:          100BASE-T1 / 1000BASE-T1
```

## 4. CAN FD と Ethernet TSN の使い分け

### 4.1 特性比較

| 項目 | CAN FD | Ethernet + TSN |
|---|---|---|
| 最大速度 | 8Mbps（要確定） | 100Mbps〜1Gbps |
| バストポロジー | バス型（マルチドロップ） | スター型（スイッチ経由） |
| 遅延 | 数百μs〜ms | TSNで数十〜数百μs（要確定） |
| 接続ノード数 | 〜30ノード程度 | 実質無制限（スイッチ追加） |
| 配線コスト | 低（単純なツイストペア） | やや高（スイッチ・ハーネス） |
| 既存資産との互換 | J1939との親和性高 | CAN資産の流用不可 |
| 機能安全対応 | 確立された知見多い | TSN+機能安全は知見蓄積途上 |
| ツール・デバッガ | 成熟（Vector・PEAK等） | 成熟度向上中 |

### 4.2 建設機械向け推奨構成（2026年時点）

```
[油圧制御ECU] ─── CAN FD（J1939ベース） ─── [メインコントローラ]
[走行制御ECU] ─── CAN FD（J1939ベース） ─┘       │
[アーム制御ECU] ── CAN FD（J1939ベース） ─────────┘
                                                    │ Ethernet（100BASE-T1/TSN）
                                         [Linuxゲートウェイ ECU]
                                                    │
                                         [HMI / クラウド通信]
```

- **安全クリティカル制御系**：CAN FD + J1939ベース（Classic AUTOSAR管理）
- **情報系・ゲートウェイ**：Ethernet（TSNは段階導入）
- **将来**：制御系もEthernet TSN化（スキル・ツール・認証事例が揃ってから）

## 5. 委託先との仕様協議ポイント

### 5.1 CAN FD移行時の確認事項

- [ ] ECU上のCAN FDコントローラ対応有無（既存ハードウェアの確認）
- [ ] バス速度の設定値（アービトレーション・データフェーズ）
- [ ] クラシックCAN混在ノードの有無と段階移行計画
- [ ] J1939-17（CAN FD対応）の採用可否
- [ ] CANトランシーバの対応（TJA1044等、要ベンダー確認）

### 5.2 Ethernet TSN導入時の確認事項

- [ ] TSNスイッチの選定（Marvell 88Q5050等、要確定）
- [ ] gPTP（IEEE 802.1AS）のグランドマスター設計
- [ ] GCL（Gate Control List）のスケジューリング設計ツール
- [ ] SOME/IPとTSNの組み合わせ検証実績
- [ ] ISO 25119との整合（TSN + 機能安全の設計手法）
- [ ] AutosarのEth関連モジュール（EthIf・SoAD・SOME/IP）の設定

## 6. まとめ・移行ステップ

| フェーズ | 期間 | 内容 |
|---|---|---|
| Phase 1 | 〜2027年 | 既存CANをCAN FD対応に段階移行。J1939維持しながらデータ長・速度を拡張 |
| Phase 2 | 2027〜2029年 | ゲートウェイ・HMI間にEthernet導入（TSNなし or CBS程度） |
| Phase 3 | 2029年以降 | 制御系EthernetへのTSN適用（スキル・ツール成熟を確認してから） |

**キーメッセージ**：CAN FDは「今すぐできるCAN高速化」、Ethernet TSNは「将来のアーキテクチャ基盤」。段階的に移行することでリスクとコストを分散させる。
