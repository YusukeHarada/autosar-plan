# 診断通信（UDS / J1939）設計メモ

**作成日：** 2026年6月16日  
**対象読者：** 技術担当者・アーキテクト  
**資料種別：** 社内検討用（社外秘）

---

## 1. 診断通信の全体像

| プロトコル | 規格 | 主な用途 |
|-----------|------|---------|
| OBD-II | ISO 15031 / SAE J1979 | 排気ガス規制対応（乗用車） |
| UDS | ISO 14229 | 汎用ECU診断・書き込み・セキュリティ |
| J1939診断 | SAE J1939-73 | 商用車・建設機械の障害コード通知 |

建設機械では **UDS**（整備ツールからのオンデマンド診断）と **J1939診断**（DM1/DM2によるブロードキャスト通知）が共存する。遠隔監視（テレマティクス）はDM1のデータをゲートウェイ経由でクラウドへ送るケースが多い。

---

## 2. UDS（ISO 14229）主要サービス

### 通信フロー（CAN上 / ISO 15765 TP）

```
[診断ツール]  -- SingleFrame or FirstFrame -->  [ECU]
[ECU]         <-- FlowControl --                [診断ツール]
[ECU]         -- PositiveResponse (0x40+SID) -- [診断ツール]
              または NegativeResponse (0x7F)
```

ISO 15765（CANTP）がペイロード8バイト超のメッセージをフレーム分割する。UDSはそのアプリケーション層として乗る。

### 主要サービス詳細

#### 0x10 DiagnosticSessionControl（セッション切替）

**何のために使うか：**
UDSはセッション状態によって使用できるサービスを制限する。デフォルトセッションでは最低限の読み出ししかできず、診断・書き込み・フラッシュ書き込みには各セッションへの明示的な切替が必要。セキュリティを多層に分けることで、整備士・開発者・生産ラインのツールごとに操作範囲を分離できる。

**セッション種別：**

| セッション | サブ値 | 有効なサービス | 典型的な使用者 |
|---|---|---|---|
| DefaultSession | 0x01 | 0x22読出・0x19DTC読出など最小限 | 常時（起動直後） |
| ExtendedDiagnosticSession | 0x03 | 0x27セキュリティ解除・0x2E書込・0x85DTC停止等 | 整備士のサービスツール |
| ProgrammingSession | 0x02 | 0x34〜0x37ファームウェア書込 | OTAエージェント・生産ラインツール |

**建設機械での具体的な使用例：**
- 整備士がサービスツールを接続した際に自動的にExtendedSessionに遷移
- OTAパッケージ適用時、ゲートウェイからECUへProgrammingSessionへの遷移を指示してからフラッシュ書込を開始
- セッションタイムアウト（S3タイマー、通常5秒）を超えると自動的にDefaultSessionに戻る。フラッシュ書込中はタイマーリセットのためのTesterPresentを定期送信する必要がある

---

#### 0x27 SecurityAccess（セキュリティアクセス）

**何のために使うか：**
パラメータ書込やフラッシュ書込は、不正なツールやサイバー攻撃から保護する必要がある。シード（乱数）とキー（計算値）の照合によって「正規のツールであること」を確認してからロックを解除する。

**シーケンス：**
```
診断ツール                         ECU
     │                              │
     │ 0x27 01（シード要求）         │
     │─────────────────────────────>│
     │                              │ 乱数シードを生成
     │ 0x67 01 <シード値>            │
     │<─────────────────────────────│
     │                              │
     │ キー = SHA256(シード + 秘密鍵)│
     │ 0x27 02 <キー値>              │
     │─────────────────────────────>│
     │                              │ 同じ計算をECU内で実施
     │ 0x67 02（解除成功）           │ 一致 → ロック解除
     │<─────────────────────────────│
```

**建設機械での具体的な使用例：**
- 油圧リリーフ圧・エンジン出力制限値などの調整パラメータを書き込む前に必須
- 生産ラインでのEOLキャリブレーション値書込時（0x2E）の前処理として使用
- シードは起動ごとに乱数生成（固定シードは禁止）。失敗3回でdelayTimer（秒単位）ロック

---

#### 0x22 ReadDataByIdentifier（データ識別子による読出）

**何のために使うか：**
ECU内のセンサー値・設定値・バージョン情報・累積稼働カウンタなどを、データ識別子（DID: 2バイト）で指定して読み出す。診断の第一歩として最も頻繁に使うサービス。

**建設機械での具体的な使用例：**

| DID（例） | データ内容 | 典型的な用途 |
|---|---|---|
| 0xF187 | ECUソフトウェアバージョン | OTA後のバージョン確認 |
| 0xF18C | ECUシリアル番号 | 整備記録との紐付け |
| 0xF190 | VIN（機体識別番号） | フリート管理システムとの照合 |
| 0x2001（メーカー定義） | 現在の作動油温度 | 過熱診断・暖機状態確認 |
| 0x2002（メーカー定義） | エンジン累積稼働時間 | 定期整備トリガーの確認 |
| 0x2010（メーカー定義） | 現在のエンジン出力制限設定値 | 出荷前キャリブレーション確認 |

複数DIDを1リクエストにまとめて読み出すことが可能（ISO 14229の仕様上最大253バイトまで）。CANoeのDiag Basicスクリプトで定期ポーリングを組めば、テスト中のパラメータ変化を自動記録できる。

---

#### 0x2E WriteDataByIdentifier（データ識別子による書込）

**何のために使うか：**
キャリブレーション値・設定パラメータ・ラベル情報をECUに書き込む。ExtendedSession + SecurityAccess解除後に使用可能。書込後は必ず0x22で読み返して整合性を確認すること。

**建設機械での具体的な使用例：**
- 生産ライン終端（EOL）でのシリアル番号・キャリブレーション値書込
- 油圧ポンプ交換後のポンプ固有補正値の更新（ポンプごとに効率特性が微妙に異なるため）
- 地域設定（排ガス規制・エンジン出力制限）の書込（輸出仕向け変更時）

---

#### 0x85 ControlDTCSetting（DTC記録の停止/再開）

**何のために使うか：**
テストやキャリブレーション作業中に意図的に異常状態を作り出す場合、DTCが誤記録されてしまうのを防ぐ。作業前に記録を停止し、作業後に再開する。

**建設機械での具体的な使用例：**
- 油圧シリンダーのストロークセンサーを意図的に外してキャリブレーション作業をする際、センサー断線DTCが発生するのを防ぐ
- HILテスト環境で異常条件を注入する際に不要なDTC蓄積を避ける
- サービスマニュアルの整備手順に「0x85で停止 → 作業 → 0x85で再開 → 0x22で状態確認」を明記することを推奨

**注意：** 0x85の停止状態でシャットダウンすると次回起動でも停止状態が続く実装と、DefaultSessionに戻ると自動再開する実装がある。Dcmのコンフィグで明示的に挙動を定義すること。

---

#### 0x14 ClearDiagnosticInformation と 0x19 ReadDTCInformation

**0x19 ReadDTCInformation**：
- サブ機能でフィルタ条件を指定してDTCを読み出す（確認済み・未確認・特定ステータスビット等）
- スナップショットデータ（DTC発生時の各センサー値）もここで取得できる

**0x14 ClearDiagnosticInformation**：
- 0xFFFFFF（全グループ）または特定のDTCグループを指定して消去
- Dem_ClearDTCが呼ばれ、NvMブロックが消去される

**建設機械での具体的な使用例：**
- 整備完了後に0x14でDTCをクリアし、試運転で再発しないことを0x19で確認
- テレマティクスGW経由でクラウドにDTCログをアップロードする前に0x19で完全なスナップショットを収集

---

#### 0x34 / 0x36 / 0x37 RequestDownload / TransferData / RequestTransferExit（ファームウェア書込）

**何のために使うか：**
ECUのフラッシュメモリにファームウェアイメージを転送・書き込む。ProgrammingSession + SecurityAccess解除後に使用。OTAとサービスツールによる手動フラッシュの両方で使われる。

**建設機械での具体的な使用例：**
- OTAエージェントがLinuxゲートウェイ上でイメージを受信後、UDS 0x34→0x36→0x37のシーケンスでECUにフラッシュを転送
- A/Bパーティションのどちらに書き込むかはECU側のBootloaderが制御。書込完了後に0x11（ECUReset）でリセットし、新バージョンから起動して整合性チェック

## 3. AUTOSARのDcm・Demモジュール

**Dcm（Diagnostic Communication Manager）**
- UDSサービスの受信・解釈・ルーティングを担う
- セッション状態機械・セキュリティ状態機械を内部で管理
- 各サービスハンドラはアプリ層のコールバック（`Xxx_GetDataElement` 等）を呼び出す

**Dem（Diagnostic Event Manager）**
- アプリ・BSWからのイベント（`Dem_ReportErrorStatus`）を受け付けDTCに変換
- DTCのステータスビット（TestFailed / Confirmed / Pending 等）を管理
- **NvMとの連携**：Dem は NvM ブロックにDTCステータス・フリーズフレームを書き込む。シャットダウン時に`NvM_WriteAll`で永続化され、次回起動時に復元される

---

## 4. DTCのライフサイクル

DTCは「発生→確認→通知→消滅→消去」という明確なライフサイクルをたどる。各遷移条件を正しく設計しないと、「消えないDTC」や「すぐに消えるDTC」が発生し、整備品質に影響する。

```
         アプリ/BSWが Dem_ReportErrorStatus(FAILED) を呼ぶ
                           │
                           ▼
              ┌─────────────────────┐
              │  TestFailed セット   │  ← 1回のイベント発生で即座にセット
              │  Pending セット      │  ← 同一ドライビングサイクル内
              └──────────┬──────────┘
                         │
              閾値チェック（Confirmed Threshold）
              デフォルト：2回失敗で確認済みに昇格
                         │
                         ▼
              ┌─────────────────────┐
              │    Confirmed セット   │  ← DTCが「確認済み障害」として記録
              │    DM1（J1939）送信  │  ← J1939変換レイヤが読み出し
              │    警告灯点灯トリガ  │  ← FiM経由で機能停止も可能
              └──────────┬──────────┘
                         │
              イベント消滅（PASSED報告）
                         │
                         ▼
              ┌─────────────────────┐
              │   Healing カウンタ   │  ← PASSED連続N回で回復判定
              │   TestFailed クリア  │
              │（Confirmedは残る）   │  ← 過去の記録として保持
              └──────────┬──────────┘
                         │
              整備士が 0x14 ClearDiagnosticInformation を送信
                         │
                         ▼
              ┌─────────────────────┐
              │  Dem_ClearDTC 呼出   │
              │  NvMブロック消去     │  ← 次回NvM_WriteAllで永続消去
              │  全ステータスビット  │
              │  クリア              │
              └─────────────────────┘
```

### ステータスビットの詳細

| ビット | 名称 | 意味 | 建設機械での注意点 |
|---|---|---|---|
| bit0 | TestFailed | 現在のテストが失敗中 | 走行中に変化しうる。DM1の基準になる |
| bit1 | TestFailedThisDrivingCycle | 今回の稼働サイクルで失敗した | キーOFF→ON でクリア |
| bit2 | PendingDTC | 未確認（閾値未達）の障害 | 閾値を超えるとConfirmedに昇格 |
| bit3 | ConfirmedDTC | 確認済み障害 | 0x14でのみクリア可能 |
| bit4 | TestNotCompletedSinceLastClear | 前回クリア後に未テスト | 整備後の確認走行が必要かどうかの判断に使う |
| bit5 | TestFailedSinceLastClear | 前回クリア後に失敗あり | 整備の効果確認に使う |
| bit6 | TestNotCompletedThisDrivingCycle | 今回稼働サイクルで未テスト | 稼働条件（温度・回転数等）が揃っていないと更新されない |
| bit7 | WarningIndicatorRequested | 警告灯点灯要求中 | 運転者への通知状態の管理 |

### フリーズフレーム（スナップショット）

DTCがConfirmedに昇格した瞬間のセンサー値一式をフリーズフレームとして記録する。建設機械では以下を記録対象とすることを推奨する：

- エンジン回転数・負荷率
- 作動油温度・冷却水温度
- 油圧（メイン回路・パイロット回路）
- 機体姿勢（IMU値）
- 累積稼働時間
- 直前の操作入力（ジョイスティック指令値）

## 5. J1939診断（DM1/DM2）との使い分け

| | DM1 | DM2 |
|--|-----|-----|
| 内容 | アクティブな障害コード（SPN+FMI） | 過去の障害コード |
| 送信方式 | 1秒周期ブロードキャスト | オンデマンド（要求応答） |
| 主な受信者 | 車載クラスタ・テレマティクスGW | サービスツール |

### J1939 DM1とUDSの「どちらを使うべきか」の判断基準

建設機械では両方を実装することが多いが、用途・受信者・通信バスによって役割が明確に分かれる。以下の判断フローで設計方針を決定すること。

```
                  ┌─────────────────────────────┐
                  │ 何を実現したいか？           │
                  └──────────────┬──────────────┘
                                 │
         ┌───────────────────────┼───────────────────────┐
         │                       │                       │
         ▼                       ▼                       ▼
  ┌─────────────┐       ┌─────────────┐         ┌─────────────┐
  │ 車載クラスタ │       │ テレマティクス│         │ 整備ツール  │
  │ ・警告灯制御 │       │ ・クラウド転送│         │ ・詳細診断  │
  │ ・オペレータ │       │ ・遠隔監視   │         │ ・書込・    │
  │   通知       │       │             │         │   フラッシュ│
  └──────┬──────┘       └──────┬──────┘         └──────┬──────┘
         │                     │                        │
         ▼                     ▼                        ▼
   J1939 DM1            J1939 DM1                    UDS
   （推奨）             （推奨）                （必須・推奨）
   リアルタイム          ゲートウェイが              ISO 15765 TP
   ブロードキャスト      購読してMQTTで              専用診断バスまたは
                        クラウドへ転送              物理レイヤ分離
```

**詳細な使い分け基準：**

| 判断軸 | J1939 DM1/DM2 を選ぶ | UDS（ISO 14229）を選ぶ |
|---|---|---|
| **受信者** | 車載クラスタ・テレマティクスGW・他ECU | 整備士のサービスツール・OTAエージェント |
| **通信方式** | ブロードキャスト（全バス参加者が受信） | リクエスト/レスポンス（1対1） |
| **情報量** | SPN+FMI+発生カウンタ（最小限） | スナップショット・DTCステータス全ビット・任意のDataIDまで（詳細） |
| **整備バスへの影響** | 常時送信のため帯域消費あり | オンデマンドのみ。帯域影響小 |
| **対象CANバス** | 車体CANバス（J1939主バス） | 診断専用バスまたは同一バス上の別アドレス |
| **パラメータ書込・フラッシュ** | 非対応 | UDSのみ対応（0x2E・0x34〜0x37） |
| **セキュリティアクセス** | なし（平文ブロードキャスト） | 0x27でシード/キー認証 |
| **既存インフラとの親和性** | コマツ・日立等の既存テレマティクス基盤に多い | サービスツール（ETAS INCA・CANoe）が標準対応 |

**典型的な共存パターン（推奨構成）：**
```
[建設機械メインコントローラ]
    │
    ├─── J1939 CANバス（1Mbps）─── 車載クラスタ・テレマティクスGW・他ECU
    │       DM1（1秒周期）: 現在アクティブな障害をブロードキャスト
    │
    └─── 診断CAN / CANFD（500kbps〜2Mbps）─── 整備コネクタ（16ピンDLC）
            UDS via ISO 15765 TP: 詳細診断・書込・フラッシュ専用
```

**同一CANバスで共存させる場合の注意：**
J1939とUDSを同一CANバスで動かすことも技術的には可能だが、J1939アドレス（8ビット）とUDSのTPアドレス（0x7xx）の競合を避けるためにアドレス設計を慎重に行うこと。帯域消費も増えるため、大型建設機械（CANバス数が多い機種）では専用診断バスを持つことを強く推奨する。

## 6. UDSとJ1939の連携実装（AUTOSARスタック内）

AUTOSAR Demが内部でDTCを管理し、J1939変換レイヤがそれを読み出してDM1メッセージを生成する。

```
アプリSWC
    │  Dem_ReportErrorStatus(FAILED)
    ▼
  Dem（DTCステータス管理・NvM保存）
    │                    │
    │ 0x19サービス        │ J1939変換レイヤ（ComM/ComM拡張またはカスタムSWC）
    ▼                    ▼
  Dcm（UDS応答）       DM1メッセージ生成
    │                    │
    ▼                    ▼
診断ツール            J1939 CANバス → クラスタ・GW
```

## 7. 設計上の注意点

**セキュリティアクセス（0x27）**
- シードは起動ごとに乱数生成。固定シードは禁止
- キーアルゴリズムは量産ECUとサービスツールで同一ライブラリを使い乖離を防ぐ
- 失敗回数上限（デフォルト3回）を超えたら一定時間ロック（`securityAccessDelayTimer`）

**EOL（End of Line）テスト**
- 生産ラインでExtendedセッションに入り、0x2Eでシリアル番号・キャリブレーション値を書込
- 書込後に0x22で読み返し値の整合性を確認

**OTAとの連携**
- テレマティクスGWがOTAサーバーからパッケージを受信後、UDS 0x34〜0x37シーケンスでECUにフラッシュを転送するトリガーとして診断セッションを使用
- Programming Sessionへの遷移前に0x27でセキュリティ解除が必須

## 8. PoCとしてDcm+Demを選ぶ理由

- **合否判定が容易**：ISO 14229はサービスIDごとのリクエスト/レスポンスフォーマットが明確に規定されており、CANoeのDiag Basicスクリプトで自動テストスクリプトを記述しやすい
- **CANoeで自動化可能**：CAPL + Diagnostic Description（.cdd/.pdx）を使えば、UDSシーケンス全体を回帰テストとして実行できる
- **BSW間インタフェースの検証範囲が広い**：DcmはCom/PduR/CanTp、DemはNvM/FimなどAUTOSARスタックの主要モジュールと連携するため、統合検証のカバレッジが高い
- **建設機械固有要件との照合**：J1939 DM1との二重管理（DemがDTCを保持しJ1939変換レイヤが読み出す）もPoC段階で設計検証できる

## 9. 診断トランスポート層の概要（DoCAN・DoIP）

UDS（ISO 14229）はアプリケーション層のプロトコルであり、それ自体はメッセージを運ぶ仕組みを持たない。UDSメッセージをECUとツール間で運ぶ「トランスポート層」が別途必要になる。現在、建設機械で現実的に使われる、または将来使われるトランスポート層はDoCAN（CAN上）とDoIP（Ethernet上）の2つである。

---

### DoCAN（Diagnostics over CAN）

**正式名称：** ISO 15765-2（CAN Transport Protocol、通称CanTp）

**何のために使うか：**
CANフレームのペイロードは最大8バイトしかない。一方、UDSのメッセージ（例：ファームウェアデータの転送）は数百〜数MBにおよぶ。DoCANはUDSメッセージを複数のCANフレームに分割・再組立てするトランスポート層であり、これがなければCAN上でUDSは使えない。

**フレーム種別：**

| フレーム種別 | 略称 | 役割 |
|---|---|---|
| SingleFrame | SF | ペイロード7バイト以内に収まる場合に使用。1フレームで完結 |
| FirstFrame | FF | 長いメッセージの先頭フレーム。総バイト数をヘッダに含む |
| ConsecutiveFrame | CF | FirstFrame以降の続きのフレーム。シーケンス番号で順序管理 |
| FlowControl | FC | 受信側が送信側へ「続きを送っていいか」を通知する制御フレーム |

**マルチフレーム転送のシーケンス図：**

```
診断ツール（テスター）                         ECU
        │                                      │
        │ ── FirstFrame（FF） ────────────────>│  メッセージ開始・総サイズ通知
        │                                      │
        │ <── FlowControl（FC: ContinueToSend）│  「続きを送れ」と応答
        │                                      │
        │ ── ConsecutiveFrame #1（CF SN=1） ──>│
        │ ── ConsecutiveFrame #2（CF SN=2） ──>│
        │ ── ConsecutiveFrame #3（CF SN=3） ──>│  ブロック送信完了
        │                                      │
        │ <── FlowControl（FC: ContinueToSend）│  次ブロックを許可
        │ ── ConsecutiveFrame #4（CF SN=4） ──>│
        │          （以降繰り返し）             │
        │                                      │ メッセージ再組立て完了
        │ <── UDS PositiveResponse ────────────│
```

FlowControlにはブロックサイズ（BS: 一度に送れるCFの数）と最小間隔（STmin: フレーム間の最小送信間隔）が含まれる。ECUの受信バッファ能力に合わせてこれらを調整しないとオーバーフローが発生する。

**Classic AUTOSARでのモジュール位置：**

```
[Dcm] ── [PduR] ── [CanTp] ── [CanIf] ── [CAN Driver] ── CANバス
```

CanTp（Can Transport Protocol）がISO 15765-2の実装体。PduRouterがDcmとCanTpの間でPDUをルーティングする。

**建設機械での現実的な使用場面：**
- 整備士がサービスツール（ノートPC＋CANインターフェース）を16ピンDLCコネクタに接続してUDS診断を実施する際の標準トランスポート
- 生産ラインEOLテストでのキャリブレーション値書込（0x2E）とフラッシュ書込（0x34〜0x37）
- 現時点（2026年）ではほぼすべての建設機械の診断インターフェースがDoCAN前提

**CAN FD対応：**
ISO 15765-2:2016以降でCAN FDに対応し、1フレームのペイロードが最大64バイトに拡張された。CAN FDを採用している場合はFirstFrame不要でSingleFrameのまま送れるメッセージが増え、小〜中規模のキャリブレーション書込ではシーケンスが大幅に短縮される。ただしCanTpのコンフィグで`CanTpRxPaddingActivation`や`CanTpTxPaddingActivation`の設定が必要になる。CAN FD対応ツールとのインターフェース仕様は要確定。

### DoIP（Diagnostics over IP）

**正式名称：** ISO 13400

**何のために使うか：**
CANベースのDoCANは帯域が最大1Mbps（CAN FDで最大8Mbps）に限られ、大容量ファームウェアの転送や複数ECUの同時診断には時間がかかりすぎる。DoIPはEthernet（100BASE-TX以上）上でUDSメッセージをTCP/UDPで運ぶことで、高速・多重化された診断通信を実現する。OTAでの大容量フラッシュや工場EOLラインのタクトタイム短縮に直結する。

**プロトコル構造：**

```
[UDSメッセージ（アプリ層）]
         │
         ▼
[DoIPペイロード（DiagnosticMessage）]
         │
         ▼
[TCP / UDP（ポート13400）]
         │
         ▼
[Ethernet（100BASE-TX / 1000BASE-T）]
```

UDP:13400はDoIPエンティティの探索・アナウンスに使う。TCP:13400は診断メッセージ（UDS）の転送に使う。

**DoIPゲートウェイとECUの関係：**

```
[診断ツール / OTAサーバー]
         │ TCP:13400（DoIP）
         ▼
[DoIPゲートウェイ（LinuxゲートウェイECU または 専用GW）]
  ┌──────┴───────┐
  │  DoIPエンティティ │  ← ISO 13400 Edge Node
  │  論理アドレス管理 │  ← 各ECUをアドレスで識別
  └──────┬───────┘
         │ UDS via CanTp（CAN/CAN FD）
         ├─── ECU-A（エンジンコントローラ）
         ├─── ECU-B（油圧コントローラ）
         └─── ECU-C（アームコントローラ）
```

DoIPゲートウェイはEthernet側でDoIPを受け、CAN側でDoCANに変換してターゲットECUにルーティングする。ゲートウェイ自身もDoIP Entityとして論理アドレスを持ち、ツールはアドレス指定で任意のECUを直接診断できる。

**DoCAN vs DoIPの比較：**

| 項目 | DoCAN | DoIP |
|---|---|---|
| **規格** | ISO 15765-2 | ISO 13400 |
| **物理層** | CAN / CAN FD | Ethernet（100M以上） |
| **帯域** | 最大1Mbps（CAN）/ 最大8Mbps（CAN FD） | 100Mbps〜1Gbps |
| **最大フレームサイズ** | 8バイト（CAN）/ 64バイト（CAN FD） | 理論上は65535バイト |
| **1MBファームウェア転送時間（概算）** | 約10〜30秒（CAN、要確定） | 約1〜3秒（100M Ethernet、要確定） |
| **同時診断ECU数** | 基本的に1対1 | 複数ECUへの同時・並列送信が可能 |
| **ツール接続形態** | 16ピンDLC＋CANインターフェース | Ethernet（RJ45またはOBD拡張） |
| **インフラコスト** | 安価・既存整備ツールが使える | ECUへのEthernet実装が必要 |
| **OTA対応** | 厳しい（速度・帯域の制約） | 適している |

**建設機械での必要性：**
- 現状（2026年）の建設機械はDoCANが主体。DoIPは将来検討フェーズ
- 次世代機でSoC＋Hypervisor構成をとる場合、車載Ethernetバックボーンが前提となりDoIPが自然な選択になる
- 工場EOLラインで多数のECUを短タクトで書き込む場合、DoIPへの移行でライン生産性が向上する
- OTA（無線更新）を本格運用するには、ゲートウェイがDoIPを介してECUにフラッシュを転送するアーキテクチャが現実的

**Classic AUTOSARでのモジュール構成：**

```
[Dcm]
   │
[PduR]
   │
[DoIP（BSWモジュール）]
   │
[SoAD（Socket Adaptor）]
   │
[TcpIp]
   │
[Ethernet Driver]
   │
Ethernet PHY
```

DoIPモジュール自体はAutosar BSWの標準モジュールとして規定されているが、ベンダー実装の成熟度にばらつきがある。採用前にベンダーの実績を確認すること（要確定）。

## 10. SOVD（Service-Oriented Vehicle Diagnostics）

**何のために使うか：**
従来のUDS診断はリクエスト/レスポンス型の「命令式」インターフェースであり、専用の診断ツールと独自のDiagnostic Description（ODXファイル）がなければ使えない。SOVDはこの診断機能をRESTful API（HTTP/JSON）として公開し、クラウド・スマートデバイス・汎用アプリケーションから直接診断データを扱えるようにする概念である。診断インターフェースがWebサービス化されることで、ツール依存性が下がり、クラウド連携・リモート診断・OTAとの統合が大幅に容易になる。

**正式名称と関連規格：**
- SOVDは現在規格化途上であり、HIS（ドイツOEMフォーラム）やOEMフォーラム発の仕様をベースにISO 22900-4として規格化が進んでいる（要確定）
- 診断データ記述はODX（Open Diagnostic data eXchange、ISO 22901-1）、診断テストシーケンスはOTX（Open Test sequence eXchange、ISO 22901-2）が従来から規定されており、SOVDはこれらの上位に位置づけられるアクセス層

**従来UDS vs SOVDの比較：**

| 観点 | 従来UDS（DoCAN / DoIP） | SOVD |
|---|---|---|
| **プロトコル** | ISO 14229（バイト列のリクエスト/レスポンス） | HTTP / REST（JSON） |
| **インターフェース** | バイナリフォーマット（SID+サブ機能） | HTTPエンドポイント（GET/POST） |
| **ツール依存性** | 専用診断ツール＋ODXファイルが必須 | 汎用HTTPクライアントで扱える |
| **クラウド連携** | ゲートウェイでのプロトコル変換が必要 | クラウドからREST APIで直接アクセス可能 |
| **セキュリティ** | SecurityAccess（シード/キー） | TLS＋OAuth2等（Web標準セキュリティ） |
| **データの可読性** | バイト列・ビット演算の知識が必要 | JSONで人間可読 |
| **成熟度** | 成熟・建設機械でも実績多数 | 規格化途上・量産実績は限定的 |

**SOVDのエンドポイントイメージ：**

```
# DTCの一覧を取得（従来: UDS 0x19 ReadDTCInformation相当）
GET /diag/v1/ecu/engine-controller/dtcs

# レスポンス（JSON）
{
  "dtcs": [
    {
      "code": "P0190",
      "status": "confirmed",
      "description": "燃料レール圧力センサー回路（フィードバック不正）",
      "snapshot": {
        "engineRpm": 1200,
        "oilTemp": 85.3,
        "timestamp": "2026-06-17T08:32:11Z"
      }
    }
  ]
}

# パラメータ書込（従来: UDS 0x2E WriteDataByIdentifier相当）
POST /diag/v1/ecu/engine-controller/parameters/outputLimit
Content-Type: application/json
{ "value": 95, "unit": "percent" }
```

**Adaptive AUTOSARとの関係（ara::diag）：**

Adaptive AUTOSARにはSOVDを実現するための`ara::diag`サービスインターフェースが規定されている。Adaptive AUTOSAR上のAA（Adaptive Application）が`ara::diag`を通じてHTTPエンドポイントを公開し、外部からRESTで診断を受け付ける構成が想定されている。

**Classic側との共存パターン：**

```
[クラウド / スマートデバイス]
         │ HTTPS REST（SOVD）
         ▼
[LinuxゲートウェイECU（Adaptive AUTOSAR or Linux OSSスタック）]
  ┌─────────────────────────────┐
  │  SOVDプロキシ               │  ← REST APIを受け付けてUDSに変換
  │  ara::diag / HTTPサーバー   │
  └────────────┬────────────────┘
               │ UDS via DoIP（TCP:13400）
               ▼
  [Classic AUTOSAR ECU群]
  Dcm / Dem / CanTp が通常通り動作
```

LinuxゲートウェイがDoIPゲートウェイを兼ねる構成であれば、ゲートウェイ上にSOVDプロキシを追加するだけで、Classic AUTOSAR側のECUを改修することなくクラウドからのRESTアクセスを実現できる。これは建設機械においてClassic側とAdaptive側の段階的共存を進める上で現実的なアプローチである。

**建設機械への適用展望：**
SOVDが実用化されると、テレマティクスGWからクラウドが直接REST APIで診断データ（DTC・センサー値・バージョン情報）を取得できる将来像が実現する。現在はJ1939 DM1をゲートウェイ経由でMQTT転送する構成が主流だが、SOVDではクラウド側のアプリケーションが「必要なデータを必要なときに」引き出すことができ、テレマティクスデータの柔軟性・粒度が大幅に向上する。また、フリート管理システムが直接OTAトリガーを発行する統合アーキテクチャも見えてくる。

**現状（2026年）の評価：**
- 規格化途上であり、量産実装例は高付加価値乗用車（プレミアムOEM）の先行開発が中心
- 建設機械での実装は2030年以降の次世代機での検討項目と位置づけるのが現実的
- 現段階でSOVDを視野に入れるとすれば、LinuxゲートウェイのアーキテクチャをSOVDプロキシ追加に対応できる拡張性を確保しておくことが有効

建設機械の設計段階で「どの診断プロトコルを使うべきか」を判断するための選択フローを示す。

### 選択フロー

```
┌──────────────────────────────────────────────────────┐
│ 診断の目的は何か？                                    │

                        │
       ┌────────────────┼──────────────────┬───────────────────────┐
       ▼                ▼                  ▼                       ▼
 ┌──────────┐    ┌──────────────┐   ┌────────────────┐   ┌──────────────────┐
 │ 車載通知  │    │ 整備・詳細診断│   │ FW書込・OTA    │   │ クラウド直接診断  │
 │ 警告灯制御│    │ パラメータ確認│   │ EOLフラッシュ  │   │ テレマティクス連携│
 └────┬─────┘    └──────┬───────┘   └──────┬─────────┘   └────────┬─────────┘
      │                  │                  │                       │

 （推奨）            （現在主流）        （現在主流）           または
 J1939-73                                  ↓                 J1939→MQTT変換

```

### 建設機械における各プロトコルの現在地

| プロトコル | 建設機械での現状（2026年） | 推奨フェーズ |
|---|---|---|
| **J1939 DM1/DM2** | 主流・既存機に広く実装済み | 現在〜継続して使用 |
| **UDS + DoCAN** | 整備ツール・EOLでの標準。新規開発機から順次導入 | 現在〜必須 |
| **UDS + CAN FD（CanTp拡張）** | CAN FD採用機で有効。フラッシュ時間短縮に貢献 | 現在〜CAN FD移行と連動 |
| **UDS + DoIP** | Ethernet搭載次世代機で検討フェーズ | 2027〜2029年以降（要確定） |
| **SOVD** | 実装例なし・規格化途上 | 2030年以降（次世代機） |

### 判断マトリクス

| 判断軸 | J1939診断 | UDS / DoCAN | UDS / DoIP | SOVD |
|---|---|---|---|---|
| **整備士の手動診断** | 不向き（情報量不足） | 最適 | 将来有効 | 将来有効 |
| **車載クラスタへの障害通知** | 最適 | 非対応 | 非対応 | 非対応 |
| **テレマティクス（クラウドへのDTC送信）** | 主流（DM1をGWがMQTT転送） | 可能だが構成複雑 | 可能 | 理想的（将来） |
| **ファームウェアOTA（大容量）** | 非対応 | 低速・実用困難 | 最適 | 連携可能 |
| **工場EOLタクトタイム短縮** | 非対応 | CAN FDで改善可能 | 最適 | ─ |
| **Ethernet未搭載機（既存機）** | 使える | 使える | 使えない | 使えない |
| **導入コスト・ツール整備** | 低い（既存資産） | 中程度 | 高い（Ethernet実装が必要） | 未知数 |

### 建設機械における現在地の整理

```
                     【2026年現在の建設機械の位置】

  低機能・低コスト <────────────────────────────────> 高機能・高コスト
                                         ↓ 目指す方向
  J1939主体    ──── CAN FD移行中 ──── DoIP検討 ──── SOVD（将来）
     ★現在                ★一部先行機         将来            次世代
     （ほぼ全機）         （新型フラッグ       （2027〜         （2030〜）
                          シップ機のみ）       2029年頃）
```

**移行の基本方針：**
1. J1939 DM1によるブロードキャスト通知は継続維持（他ECU・クラスタとの互換性を保つため）
2. 新規開発機にはUDS+DoCANを標準として実装し、整備ツール対応・OTAの基盤を整える
3. CAN FD採用機ではCanTpのCAN FD拡張を活用してフラッシュ転送時間を短縮する
4. 次世代SoC搭載機で車載Ethernetを実装する際にDoIPゲートウェイを統合する
5. SOVD・ara::diagはAdaptive AUTOSAR導入の検討と並行して2028年以降に評価開始する
