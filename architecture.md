# アーキテクチャ概要

## 1. E/Eアーキテクチャの変遷と建設機械への適用

### アーキテクチャの進化

| 世代 | 構成 | 特徴 | 建設機械への適用 |
|------|------|------|-----------------|
| 分散型 | 機能ごとに専用ECU | ECU数が多く、配線複雑 | 現在の主流（既存機種） |
| ドメイン集中型 | パワートレイン／ボディ等ドメインごとにメインECU | ECU数削減、Ethernetバックボーン | 本プロジェクトの目標段階 |
| ゾーン型 | 車体ゾーンごとにゾーンECU + 中央コンピュータ | 完全SDV化 | 将来（5〜10年後） |

建設機械は自動車より開発サイクルが長く、部品調達制約も大きい。まずドメイン集中型を外部委託で確立し、段階的にSDV化を進める。

## 2. 推奨アーキテクチャ構成

### 現在：Classic AUTOSAR + Linux Gateway（物理分離）

```
┌─────────────────────────────────────────────────────────┐
│                    Vehicle Network                        │
│                                                           │
│  ┌──────────────────┐        ┌──────────────────────┐   │
│  │  Classic AUTOSAR │  CAN FD│   Linux Gateway ECU  │   │
│  │  Safety ECU      │◄──────►│                      │   │
│  │  (油圧制御/走行) │        │  SOME/IP ↔ CAN 変換  │   │
│  │  ASIL-D          │        │  TCP/IP スタック      │   │
│  └──────────────────┘        │  Fleet Management    │   │
│                               └──────────┬───────────┘   │
│  ┌──────────────────┐                    │ Ethernet       │
│  │  J1939 Network   │                    ▼               │
│  │  (エンジン/油圧) │             ┌──────────────┐       │
│  │  既存機器との共存│             │  クラウド接続 │       │
│  └──────────────────┘             └──────────────┘       │
└─────────────────────────────────────────────────────────┘
```

### 将来：SoC Hypervisor統合

Classic AUTOSARとLinuxを単一SoC上のHypervisorで分離実行。物理ECU数を削減しつつ、安全ドメインのアイソレーションを維持する（詳細は `hypervisor.md` 参照）。

## 3. SOME/IPゲートウェイアーキテクチャ

```
Classic AUTOSAR側                   Linux側
──────────────────────────────────────────────────────
  CAN FD Frame                       SOME/IP Message
  [ID | DLC | Data]                  [Header | Payload]
        │                                   ▲
        ▼                                   │
  COM / IPDU                         vsomeip / CommonAPI
        │                                   │
        ▼                                   │
  ┌─────────────────────────────────────────────────┐
  │              Gateway ECU                         │
  │  ┌─────────────────┐   ┌──────────────────────┐ │
  │  │  AUTOSAR COM    │   │  Signal Router       │ │
  │  │  (受信・解析)   │──►│  (マッピング・変換)  │ │
  │  └─────────────────┘   └──────────────────────┘ │
  └─────────────────────────────────────────────────┘
```

ゲートウェイECUはシグナル変換のみを担う薄い層とし、ビジネスロジックを持たない設計とする。これにより変換レイヤのテストが容易になる。

## 4. セキュリティアーキテクチャ

### 多層防御の考え方

| 層 | 技術 | 役割 |
|----|------|------|
| メッセージ認証 | SecOC（AUTOSAR） | ECU間CAN FDフレームのMAC検証 |
| 鍵管理・暗号 | HSM（オンチップ） | 秘密鍵の安全な保持、AES/SHA処理 |
| ネットワーク分離 | VLAN + ゾーンFirewall | 安全系／情報系の通信ドメイン隔離 |
| 外部接続 | TLS 1.3 + 証明書 | クラウド・診断通信の暗号化 |

SecOCはCAN FD上の全安全関連メッセージに適用する。HSMは安全ECUに内蔵し、鍵がSWに露出しない構成とする。

## 5. 建設機械固有の考慮事項

### 油圧制御系の安全アーキテクチャ

自動車のE-gas（電子スロットル）と同様に、油圧バルブ制御でも「指令経路」と「監視経路」を独立させる。

```
操作指令 ──► [主制御CPU: ASIL-C]──► 油圧バルブ
                     │
                     ▼ 監視（クロスチェック）
             [監視CPU: ASIL-D] ──► 緊急遮断弁
```

監視CPUは主制御CPUの出力を独立して検証し、異常時は油圧系統を安全状態（バルブ閉）へフェールセーフさせる。

### J1939とAUTOSARの共存

既存の油圧・エンジンECUはJ1939ベースで通信している。移行期間中は以下の方針で共存させる。

- J1939 ↔ CAN FD変換はゲートウェイECUで吸収
- AUTOSARのPDUマッピングでJ1939 PGNをシグナルとして取り込む
- 新規ECUのみClassic AUTOSARで実装し、既存ECUは原則改修しない

これにより既存機種への影響を最小化しながら、新制御機能をAUTOSAR上に段階的に移行できる。
