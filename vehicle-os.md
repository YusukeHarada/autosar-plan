## 1. CASE（Connected / Autonomous / Shared / Electric）

### 1.1 CASEとは

CASEは自動車産業が直面する4つの構造変化の頭文字をとった概念。それぞれが独立して進化するのではなく、相互に影響しながら車両・建設機械のアーキテクチャを根本から変えつつある。

| 要素 | 概要 | 建設機械への影響 |
|------|------|----------------|
| **C**onnected | 常時ネットワーク接続・データ活用 | フリート管理・稼働データ収集・OTA更新 |
| **A**utonomous | 自律走行・自動制御 | 半自律施工・人検知・障害物回避 |
| **S**hared | シェアリング・サービス化 | レンタル最適化・稼働率向上・課金モデル変革 |
| **E**lectric | 電動化・エネルギー管理 | バッテリ管理・電動アクチュエータ制御 |

### 1.2 各要素の建設機械への適用

**Connected（接続性）**  
建設機械は現状でもコマツ・CAT等がフリート管理システム（KOMTRAX・Product Link）を導入しているが、通信はGPS位置・稼働時間の収集にとどまるケースが多い。  
今後はCAN/Ethernetデータのリアルタイム収集、OTA（Over-The-Air）ソフトウェア更新、遠隔診断が中心となる。クラウド連携はLinuxゲートウェイ経由でAWS IoT Core / Azure IoT Hubに接続する構成が現実的（→ `cloud-connectivity.md` 参照）。

**Autonomous（自律化）**  
建設機械での自律化は「完全自律」ではなく「半自律施工」が現実的な近中期目標。具体的には：
- キャビン後方・側方の人検知・自動停止（安全機能）
- ティーチングプレイバックによる繰り返し作業の自動化
- ICT施工における3D設計データとの誤差自動補正

ISO 25119（農業機械・建設機械向け機能安全）への対応が不可欠であり、制御系はRTOS + Classic AUTOSARによる安全クリティカル設計が必要。

**Shared（共有・サービス化）**  
建設機械はもともとレンタル比率が高い業界。CASEのSharedはレンタル会社が稼働データをもとに最適配車・予防保全・稼働率課金（Equipment as a Service）へと転換する動きに対応する。ソフトウェアによる機能のライセンス制御（特定機能を有効化）も将来の収益モデルとなりうる。

**Electric（電動化）**  
小型機を中心に電動化が進んでいる（要確定：各メーカーの電動機ラインナップ）。電動化により：
- エンジン制御ECUが不要になりECU数削減の契機に
- バッテリ管理システム（BMS）が新たな重要ECUとして加わる
- 回生エネルギー制御のためソフトウェアの役割が拡大

### 1.3 CASEがE/Eアーキテクチャ変化を加速した背景

CASE以前の自動車・建設機械は機能ごとにECUを追加する「分散型アーキテクチャ」が主流だった。しかしCASEの4要素はいずれも大量のセンサデータ処理・高速通信・ソフトウェア更新を要求するため、分散型の限界が露呈した：

- ECU数の増大（高級車で100個超）→ ハーネス重量・コスト・複雑性の爆発
- CAN帯域の枯渇 → Ethernetバックボーンの必要性
- ソフトウェア更新の困難さ → OTAのためのアーキテクチャ整備

この背景から「ドメイン型」→「ゾーン型」→「集中型（セントラルECU）」という移行が加速している。

## 2. ゾーンアーキテクチャ（Zone Architecture）

### 2.1 ドメイン型の限界

ドメイン型アーキテクチャはパワートレイン・シャシ・ボディ・インフォテインメントなど「機能領域（ドメイン）」単位でECUを束ねる構成。車両が大型化・複雑化するにつれて以下の問題が顕在化した：

- **ハーネスの複雑化**：離れた場所のECUを機能ドメインで束ねるため、配線が車体中央に集中し長距離化
- **重量増**：建設機械では数十kgの配線束になるケースがある
- **ECU追加の困難さ**：機能を追加するたびに既存ネットワークへの接続設計が必要

### 2.2 ゾーン型への移行

ゾーン型は「物理的な位置（ゾーン）」でECUを束ねる構成。各ゾーンにZCU（Zone Control Unit）を設置し、そのゾーン内のアクチュエータ・センサをローカルに束ねる。

```
┌─────────────────────────────────────────────────────────────┐
│                    セントラルECU / HPC                       │
│              (高度演算・AUTOSAR RTE・クラウド連携)            │
└──────────────┬──────────────────────────────┬───────────────┘
               │   バックボーン Ethernet        │
      ┌────────┴──────┐              ┌──────────┴──────┐
      │  前方ZCU      │              │  後方/作業機ZCU  │
      │  (キャビン)   │              │  (ブーム・バケ)  │
      └────┬──────────┘              └──────────┬───────┘
           │ ローカルCAN/LIN                     │ ローカルCAN
    ┌──────┼──────┐                    ┌─────────┼──────┐
  HMI   カメラ  人検知               油圧弁  角度センサ  荷重
  ECU   ECU    ECU                  ECU    ECU       ECU
```

| 通信区分 | プロトコル | 帯域 | 用途 |
|----------|-----------|------|------|
| ゾーン内 | CAN / CAN FD / LIN | 低〜中 | センサ・アクチュエータ間の制御 |
| ゾーン間 | Ethernet（100BASE-T1 / 1000BASE-T1） | 高 | ZCU間・セントラルECU間の高速通信 |

### 2.3 建設機械への具体的なゾーン分割案

| ゾーン | 主なECU・デバイス | 安全要求 |
|--------|-----------------|---------|
| **キャビン・HMIゾーン** | ディスプレイECU・スイッチ・モニタカメラ | ASIL-B相当（ISO 25119） |
| **前方作業機ゾーン** | ブーム/アーム/バケット角度センサ・油圧弁ECU | ASIL-C〜D相当 |
| **走行ゾーン** | 走行モータドライバ・ブレーキECU・IMU | ASIL-C〜D相当 |
| **エンジン/動力ゾーン** | エンジンECU・BMS（電動機の場合）・油圧ポンプ制御 | ASIL-C相当 |

### 2.4 ゾーン型がOTA・機能追加に有利な理由

- ゾーンコントローラが標準化されたEthernetでセントラルECUと通信するため、ゾーン内ECUの変更がネットワーク全体に波及しにくい
- ソフトウェア更新をゾーン単位で段階的に展開できる（一括更新による全機停止リスクの低減）
- 新機能はセントラルECUへのソフトウェア追加として実装でき、ハードウェア変更を最小化できる

## 3. セントラルECU / HPC（High Performance Computer）

### 3.1 概念と役割

セントラルECUは車両・建設機械の「演算の中枢」として、従来分散していた機能を集約するECU。HPCとも呼ばれ、高性能SoC（System on Chip）を搭載する。

主な役割：
- センサフュージョン（カメラ・LiDAR・レーダの統合処理）
- Adaptive AUTOSARアプリケーションの実行基盤
- クラウド連携・OTAオーケストレーション
- 車両全体のサイバーセキュリティゲートウェイ

### 3.2 SoC選定の考え方

| SoC例 | 特徴 | 主な用途 |
|-------|------|---------|
| Qualcomm SA8295（要確定） | 車載向け・ADAS + IVI統合 | セントラルECU・HMI |
| NVIDIA Orin（要確定） | GPU演算・AI推論向け | 自律走行・センサフュージョン |
| Renesas R-Car（要確定） | 機能安全対応・ISO 26262 ASIL-B | ゲートウェイ・統合ECU |

建設機械は動作温度範囲・振動・粉塵環境が自動車より過酷であるため、車載グレード品（AEC-Q100相当）かつ広温度品の選定が必要（要確定：各SoC対応温度範囲）。

### 3.3 セントラルECUとゾーンコントローラの役割分担

```
セントラルECU（HPC）
  ├── 高度AI推論（人検知・施工自動化）
  ├── クラウド通信・OTA管理
  ├── Adaptive AUTOSAR Application実行
  └── セキュリティゲートウェイ

ゾーンコントローラ（ZCU）
  ├── リアルタイム制御（Classic AUTOSAR相当）
  ├── ローカルセンサ・アクチュエータ管理
  ├── 通信プロトコル変換（CAN ⇔ Ethernet）
  └── フェールセーフ動作（セントラルECU停止時の安全制御）
```

重要な設計原則：**ZCUは独立してフェールセーフ動作できること**。セントラルECUが故障・再起動中でも、ZCUが最低限の安全動作（緊急停止・ブレーキ保持）を継続できる設計が安全認証上必須。

### 3.4 セキュリティ考慮

セントラルECUへの集約はセキュリティリスクも集約することを意味する：

- **攻撃面の集中**：1箇所を突破されると車両全機能への影響が懸念される
- **単一障害点**：HPC故障が全機能停止に直結するリスク
- **対策**：HSM（Hardware Security Module）搭載必須・SecOC適用・ネットワーク分離（OBD接続はゾーンを分離）

### 3.5 建設機械への適用時期

現時点（2026年）では、建設機械でのセントラルECU採用は時期尚早と判断する。理由：
- Hypervisor上でのISO 25119認証実績がほぼない
- セントラルECU設計・統合スキルの社内不足
- コスト面での優位性が現状では見えにくい

**推奨適用時期：2030年代以降の次世代機設計から**（→ `roadmap.md` のフェーズ4以降）。

## 4. ビークルOS（Vehicle OS）

### 4.1 ビークルOSとは

ビークルOSは「車両向けLinuxディストリビューション＋ミドルウェア統合基盤」の総称。スマートフォンにおけるAndroid / iOSに相当する役割を、車両のソフトウェアスタックに持ち込もうとする概念。

目的：
- アプリケーション開発者がハードウェアの差異を意識せずに開発できる環境の提供
- ソフトウェアの再利用・エコシステム形成
- OTA更新のしやすいアーキテクチャ確立

### 4.2 代表的なビークルOS

| 名称 | 主体 | 特徴 |
|------|------|------|
| **Android Automotive OS（AAOS）** | Google | HMI・インフォテインメント特化。Google Playストア連携 |
| **Linux Foundation ELISA** | Linux Foundation | 安全クリティカルLinux適用のためのワーキンググループ |
| **SOAFEE（Scalable Open Architecture for Embedded Edge）** | ARM主導コンソーシアム | クラウドネイティブ技術（コンテナ・Kubernetes）の車載適用 |
| **BlackBerry QNX** | BlackBerry | RTOS。ISO 26262 ASIL-D認証実績あり |
| **VxWorks** | Wind River | 産業・航空向けRTOS。車載・建設機械での採用実績あり |

### 4.3 Classic AUTOSAR / Adaptive AUTOSARとビークルOSの関係

```
┌─────────────────────────────────────────────────┐
│              アプリケーション層                   │
│   (HMI / クラウド通信 / 機能追加アプリ)          │
├─────────────────────────────────────────────────┤
│            ビークルOS / ミドルウェア層            │
│   (Android Automotive / SOAFEE等)               │
├──────────────────────┬──────────────────────────┤
│  Adaptive AUTOSAR    │   Classic AUTOSAR         │
│  (Linux + ara::com)  │   (RTOS + BSW/RTE)        │
├──────────────────────┴──────────────────────────┤
│         HAL（Hardware Abstraction Layer）         │
├─────────────────────────────────────────────────┤
│              SoC / ハードウェア                  │
└─────────────────────────────────────────────────┘
```

Classic AUTOSARは安全クリティカルな制御系（油圧・走行・ブレーキ）を担い、ビークルOSはその上位の非リアルタイム領域（HMI・通信・AI）を担う。両者はEthernet / SOME/IPで通信する構成が自動車業界の主流となりつつある。

### 4.4 HAL（Hardware Abstraction Layer）の役割

HALはアプリケーションとハードウェアの間に入り、ハードウェア固有の処理を隠蔽する：
- SoCのCAN/Ethernetドライバの違いをHALで吸収
- カメラ・センサのインターフェース標準化
- HALが整備されることでSoCの世代交代時にアプリケーション修正を最小化

建設機械では機種・車格によりSoCが異なる場合があるため、HALの設計が保守性に直結する。

### 4.5 建設機械への適用検討

短期的（〜2028年）にビークルOSが適用可能な領域：
- **HMI・ディスプレイ**：Android Automotive OSの採用（インフォテインメント・カメラ映像表示）
- **クラウド通信・ゲートウェイ**：Linuxミドルウェア（自社開発またはYocto Projectベース）

安全クリティカル領域へのビークルOS適用は、ISO 25119対応の証跡整備が困難なため2030年代以降の検討。

## 5. オーケストレーション（Orchestration）

### 5.1 車載コンテキストでのオーケストレーションの意味

オーケストレーションとは「複数のソフトウェアコンポーネント・サービスの起動・停止・リソース配分・依存関係解決を自動的に管理すること」。

車載での具体的な意味：
- 走行開始時に必要なサービスだけを起動し、不要なサービスをシャットダウン
- 演算リソース（CPU・メモリ）を複数アプリケーションで動的に分配
- OTA更新後に新バージョンへ切り替え・問題があれば旧バージョンへロールバック

### 5.2 Adaptive AUTOSARによるオーケストレーション

Adaptive AUTOSARはオーケストレーション機能を標準として提供している：

| コンポーネント | 役割 |
|--------------|------|
| **ara::exec（Execution Management）** | Adaptive Applicationの起動・停止・状態管理 |
| **ara::sm（State Management）** | 車両状態（起動・走行・停車）に応じたサービス切り替え |
| **ara::per（Persistency）** | 設定・ログの永続化 |
| **ara::com（Communication Management）** | SOME/IPによるサービス探索・通信 |

```
State Management (ara::sm)
        │
        ▼ 状態変化通知
Execution Management (ara::exec)
        │
        ├── Application A 起動
        ├── Application B 停止
        └── Application C リソース調整
```

### 5.3 クラウドネイティブ技術の車載適用（SOAFEE）

SOAFEE（Scalable Open Architecture for Embedded Edge）は、ArmがリードするコンソーシアムでKubernetesやOCI準拠コンテナを車載エッジに適用するための仕様を策定している。

メリット：
- クラウドで開発したコンテナをそのまま車両にデプロイ可能（理想）
- CI/CDパイプラインをクラウドと統合できる
- スケーラビリティ：ハードウェアを変えても同じソフトウェアが動作

現状の課題（2026年時点）：
- 安全認証との整合性（KubernetesはISO 26262 / ISO 25119認証実績なし）
- リアルタイム性の担保（コンテナのオーバーヘッド）
- 実績のある車載製品がまだ少ない（要確定）

### 5.4 OTA・SDV化への影響

オーケストレーションが整備されることで以下が実現する：

| 機能 | 内容 |
|------|------|
| **A/Bアップデート** | 旧バージョンと新バージョンを並行保持し、問題があれば即座にロールバック |
| **フィーチャーフラッグ** | 機能を無効状態でデプロイし、有効化タイミングをクラウドから制御 |
| **動的機能追加** | 購入後にソフトウェア機能を有効化（建設機械での半自律施工機能等） |
| **ヘルスモニタリング** | 実行中アプリケーションの異常検知と自動再起動 |

### 5.5 建設機械での現実的な適用時期

| 技術 | 適用時期 | 適用領域 |
|------|----------|---------|
| OTA更新（基本） | 2026〜2028年 | Linuxゲートウェイ・HMI |
| ara::exec（Adaptive AUTOSAR） | 2028〜2030年 | ゲートウェイ・非安全系アプリ |
| コンテナオーケストレーション（SOAFEE系） | 2030年代 | HPC上のAIアプリ |
| 完全オーケストレーション（全機能） | 2032年以降 | 次世代統合アーキテクチャ |

## 6. 全体を通じた建設機械への示唆

### 6.1 2026年時点での現実的な位置付け

本ドキュメントで扱った概念群（ゾーン型・セントラルECU・ビークルOS・オーケストレーション）は、いずれも「自動車業界で進行中」または「近未来の姿」であり、建設機械での実装は5〜10年後の話が多い。

2026年現在の建設機械での最優先事項は：
1. Classic AUTOSAR準拠BSWの外部委託による整備
2. LinuxゲートウェイでのOTA基盤構築
3. 通信仕様（DBC + ArXML）の標準化

### 6.2 今から準備すること

**スキル面：**
- Ethernet車載ネットワーク（100BASE-T1・SOME/IP）の理解
- Yocto Project / Linux BSPのビルド・カスタマイズ
- コンテナ（Docker）・CI/CDの基礎（クラウドネイティブへの架け橋）

**アーキテクチャ設計の方向性：**
- ゾーン型を「目標とするアーキテクチャ」として設計指針に採用する
- 現時点のデュアルSoC（Classic AUTOSAR ECU + Linux ECU）をゾーン型への移行ステップとして位置付ける
- HALを意識した設計で、将来のSoC変更・統合に備える

### 6.3 ロードマップ上での各概念の登場時期
