# AUTOSAR移行・SDV化 ロードマップ

建設機械（RTOS系 + Linux系ハイブリッド構成）における、内製プラットフォームからAUTOSAR準拠・外部委託モデルへの移行計画。

---

## 1. 全体フェーズ概要

| フェーズ | 期間 | テーマ |
|----------|------|--------|
| Phase 1 | 今〜2年 | 土台整備：証跡・プロセス・PoC |
| Phase 2 | 2〜5年 | 移行完了：本格稼働・コネクテッド化 |
| Phase 3 | 5〜10年 | SDV化：半自律・次世代アーキテクチャ |

---

## 2. フェーズ別 詳細と技術スタック

### Phase 1：土台整備（今〜2年）

**やること**
- ISO 25119に基づくハザード分析（HARA）と安全要求の文書化
- 既存コードの機能安全証跡の整備（後付けでも記録を残す）
- Classic AUTOSAR PoCをRTOS系（油圧制御）の一部機能で実施
- CI/CDパイプライン構築（ビルド・静的解析・単体テスト自動化）
- 外部委託先の選定・契約（要求仕様・受け入れ基準の定義）

**技術選定ポイント**

| 領域 | 採用技術 |
|------|----------|
| RTOS系BSW | Classic AUTOSAR（VectorまたはDENSOTen系ベンダー） |
| Linux系MW | Yocto + 独自ミドルウェア（この段階では標準化範囲外） |
| 静的解析 | MISRA-C対応ツール（PC-lint Plus / Polyspace） |
| CI/CD | Jenkins or GitLab CI + SIL環境 |
| クラウド連携 | LinuxゲートウェイからAWS IoT Core / Azure IoT Hub（PoC） |

---

### Phase 2：移行完了（2〜5年）

**やること**
- Classic AUTOSAR本格稼働（油圧制御系の主要ECU）
- OTA基盤の構築（Linuxゲートウェイ経由でFirmware OTA）
- フリート管理システムの本番稼働（稼働時間・エラーログ収集）
- Linux系のセキュリティ強化（SELinux / SecureBoot / 証明書管理）
- サプライヤへのASIL要求を要件書に明文化

**技術選定ポイント**

| 領域 | 採用技術 |
|------|----------|
| OTA | Mender or SWUpdate（Linux系） / AUTOSAR UCM（RTOS系将来） |
| フリート管理 | AWS IoT Fleet Management / Azure IoT Hub Device Twins |
| セキュリティ | HSM活用・SecOC（AUTOSAR）・TLS 1.3（Linux） |
| テスト | HILテスト自動化・カバレッジ計測 |

---

### Phase 3：SDV化（5〜10年）

**やること**
- 人検知・周辺認識の精度向上と半自律機能の段階的搭載
- Hypervisor統合の検討・PoC（次世代機プラットフォーム、2028年以降）
- Adaptive AUTOSARの適用検討（高性能SoC搭載ECU向け）
- OTA・フリートデータを活用したフィールドフィードバックループの確立

## 3. 必要な人材・スキルセット

### 自社に置くべき人材（要求定義・受け入れ検証）

| 役割 | 人数目安 | 求めるスキル |
|------|----------|-------------|
| 機能安全マネージャー | 1〜2名 | ISO 25119 / IEC 61508の理解、HARA・FMEAの実施経験 |
| システムアーキテクト | 1〜2名 | RTOS/Linux両系統の俯瞰的設計力、外部委託管理 |
| 受け入れテスト担当 | 2〜3名 | テスト仕様策定・HIL/SIL環境の操作・合否判定 |
| クラウド/OTA担当 | 1〜2名 | AWS / Azure IoT、デバイス管理、セキュリティ基礎 |

### 外部委託先に求めるスキル

- Classic AUTOSAR BSW構築・MCAL開発実績
- MISRA-C準拠・ASPICE CL2以上のプロセス保有
- ISO 26262（自動車）または ISO 25119（農業機械）対応経験
- HILテスト環境の構築・運用能力

### 将来的に内製化を検討する領域

- アプリケーション層の制御ロジック（機械知識との融合が必要）
- フリートデータ分析・機械学習パイプライン
- OTAキャンペーン管理・ロールアウト制御

## 4. マインドセットの転換

### 「作る組織」から「定義し検証する組織」へ

これまでの内製開発では「動けばよい」が基準だった。外部委託・標準準拠への移行後は、**何を作らせるかを定義する力**と**できあがりを正しく検証する力**が自社の核心スキルになる。コードを書く量は減っても、要求仕様の質と受け入れ基準の厳密さが品質を決める。

### 技術負債と向き合う姿勢

「仕様書なし・テストなし・証跡なし」の現状は負債だが、一括解消は不可能。Phase 1では**新規開発分から正しいプロセスを適用**しつつ、既存コードは影響範囲を限定しながら段階的に置き換える。完璧を求めて動きを止めるより、記録を残しながら前進することを優先する。

### 外部標準に乗ることの意味

AUTOSARやISO 25119は制約ではなく**共通言語**である。標準に準拠することで、委託先・顧客・認証機関との対話コストが下がり、人材採用・サプライヤ交渉でも有利に働く。自社固有の「独自仕様」を減らすほど、長期的な維持コストは下がる。
