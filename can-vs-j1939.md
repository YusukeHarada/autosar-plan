# 自動車CAN vs J1939 比較整理

自動車CAN（乗用車向け）の開発経験を持つ技術者が建設機械開発に携わる際、混乱しやすい差異を中心に整理する。

---

## 1. 根本的な関係：J1939はCANの「上位層」

まず押さえておくべき前提。

```
自動車CAN（乗用車）
  物理層: ISO 11898-2（High Speed CAN）
  データリンク層: ISO 11898-1
  アプリケーション層: 各社独自（DBCで管理）

J1939（商用車・建設機械）
  物理層: ISO 11898-2（共通）
  データリンク層: ISO 11898-1（共通）
  ネットワーク層〜アプリ層: SAE J1939で標準化
```

J1939は**CANを捨てたわけではない**。物理層と基本フレーム構造はまったく同じCANを使う。異なるのは「CANのIDと8バイトデータの使い方のルール」をSAEが業界統一仕様として定めた点である。

---

## 2. フレームIDの設計思想の違い

これが最も根本的な差異。

### 自動車CAN（乗用車）

```
11bit標準フレーム例: 0x2A1
29bit拡張フレーム例: 0x18DA00F1
```

IDの意味は各社・各プロジェクトで自由に決める。「0x2A1 = エンジン回転数」はそのプロジェクト固有のルール。DBCファイルが唯一の答えであり、他社のDBCとは互換性がない。

### J1939

```
必ず29bitを使用。ID内に意味が構造化されている。

 28    26 25 24 23        16 15         8 7          0
┌────────┬──┬──┬───────────┬─────────────┬────────────┐
│Priority│ R│DP│ PDU Format │ PDU Specific│Source Addr │
│ 3bit   │1b│1b│   8bit     │    8bit     │   8bit     │
└────────┴──┴──┴───────────┴─────────────┴────────────┘
```

| フィールド | 意味 | 自動車CANでの対応概念 |
|---|---|---|
| Priority（3bit） | 送信優先度（0=最高・7=最低） | IDが小さいほど優先 → 同じ概念だが明示的 |
| DP（1bit） | PGNページ番号（0/1） | 存在しない |
| PDU Format（8bit） | メッセージ種別。PDU1（宛先あり）/PDU2（ブロードキャスト）を判定 | 存在しない |
| PDU Specific（8bit） | PDU1=宛先アドレス、PDU2=グループ拡張 | 存在しない |
| Source Address（8bit） | 送信元ECUのアドレス | 存在しない（IDからは分からない）|

**自動車CANとの最大の違い**：CANのIDを見れば「誰が（SA）・何のメッセージ種別（PF）・優先度（Priority）」が即座に分かる。自動車CANのIDはDBCなしでは意味不明だが、J1939のIDは構造を理解すれば単体で読み解ける。

## 3. メッセージ・シグナル定義の標準化

### 自動車CAN（乗用車）

- エンジン回転数の信号名・ビット位置・スケールは**プロジェクト固有**。
- A社のECUとB社のECUは同じ「エンジン回転数」でも信号定義が異なる。
- DBCファイルが「真実の源泉」。DBCなしでは信号の意味を解読できない。

### J1939

- PGN（Parameter Group Number）：メッセージを識別する番号。SAEが意味ごとに割り当て済み。
- SPN（Suspect Parameter Number）：個々のパラメータを識別する番号。SAEが割り当て済み。

```
エンジン回転数の例：
  PGN 61444 (0xF004) = EEC1（Electronic Engine Controller 1）
  SPN 190 = Engine Speed
  Resolution: 0.125 rpm/bit, Offset: 0 rpm
  → これはどのメーカのエンジンECUでも共通
```

**メリット**：コマツのエンジンECUとボッシュの油圧コントローラが同じJ1939バスに接続されていても、標準PGNに関しては互いのDBCなしで信号の意味が分かる。

**注意点**：建設機械固有の信号（バケット圧力・アーム角度等）は標準PGNに定義がないため、各社がProprietaryPGNを定義する。このProprietaryPGNについては自動車CANと同じく、DBCを共有しないと意味が分からない。

## 4. ビットレート

| | 自動車CAN（乗用車）| J1939 |
|---|---|---|
| 標準ビットレート | 500 kbps / 1 Mbps | **250 kbps** |
| 拡張 | — | 500 kbps・1 Mbps（機器依存） |

建設機械のJ1939バスは**250 kbpsが主流**。自動車開発の感覚から見ると「遅い」と感じるが、エンジン・変速機・油圧ポンプ等の制御周期（10〜1000 ms）に対して十分な帯域であり、長いバス配線（10〜40 m超）でのノイズ耐性を優先した結果。

---

## 5. アドレッシング（ネットワーク管理）

### 自動車CAN（乗用車）

- CANのID = **メッセージID**。誰が送ったかは原則IDに含まない（拡張IDで工夫するケースはある）。
- ECUのノードアドレスという概念は通常意識しない（AUTOSAR NM等では管理するが通信IDとは別）。
- IDは設計段階で静的に割り当て、ネットワークに動的に参加する仕組みはない。

### J1939

- 各ECUが**8bitのSource Address（SA）**を持つ（0x00〜0xFD）。
- **アドレスクレーム（J1939-81 NAM）**：ECUの起動時に自分のSAをネットワークへ宣言し、競合があれば調停する。プラグ＆プレイ的な動的割り当てが可能。
- SA=0xFE（Anonymous）：アドレス未取得のECUが一時的に使用。
- SA=0xFF（Global）：全ECUへのブロードキャスト宛先。

**建設機械での実態**：アタッチメント（バケット・ブレーカ等）の交換時にアドレスクレームが発生する。固定SAで運用するケースも多いが、規格上は動的を想定している点が自動車CANと大きく異なる。

## 6. Transport Protocol（8バイト超えデータ）

CANの1フレームは最大8バイト（CAN FDは64バイト）。8バイトを超えるデータを送る場合に使うTPの仕組みが異なる。

| | 自動車CAN（ISO 15765-2 / UDS TP） | J1939 TP（J1939-21） |
|---|---|---|
| 主な用途 | UDS診断通信（大容量レスポンス） | DM1の多故障コード・長いデータ送信 |
| フレーム種別 | First Frame / Consecutive Frame / Flow Control | TP.CM（接続制御）/ TP.DT（データ転送）|
| フロー制御 | あり（受信側がFlowControlで送信ペース制御） | BAMはなし、CMDTはあり |
| 接続方式 | 1対1（P2P） | BAM（ブロードキャスト）/ CMDT（P2P） |
| 最大データ長 | 4095バイト | 1785バイト（J1939-21標準） |

**BAM（Broadcast Announce Message）**：接続確立なしで一方的に分割送信する。DM1の多故障コード通知に使われる。受信側の準備ができているかを確認しない点が自動車UDS-TPと大きく異なる。

**CMDT（Connection Mode Data Transfer）**：P2Pで接続を確立してから送受信する。自動車の UDS-TPに近い概念。

## 7. 診断の仕組みの違い

自動車CANからJ1939へ移行する際に最も「別物」と感じやすい部分。

### 自動車CAN（UDS / ISO 14229）

- 診断ツールが「診断セッション開設（0x10）」→「DTC読み出し（0x19）」と**サービスIDでコマンド送受信**。
- DTCは3バイト（Status記述あり）。
- 診断ツールが主導権を持ち、ECUはレスポンスを返す。

### J1939（DM系メッセージ）

- 故障情報はECUが**自律的にPGNとして定期送信**する。
- `DM1（PGN 65226）`：現在アクティブな故障コード。ECUが1秒周期等で定期送信。
- `DM2（PGN 65227）`：過去故障コード（解消済み）。ツールからの要求（DM3）で応答。
- DTCの識別子はSPN+FMIの組み合わせ。

```
UDS DTC: 0xP0301  → エンジン失火（1番シリンダ）など16進3バイト
J1939 DTC: SPN=191（Engine Speed）+ FMI=9（更新レート異常）の組み合わせ
```

| 比較項目 | UDS（自動車） | J1939 DM |
|---|---|---|
| 故障コードの形式 | 3バイト DTC | SPN（19bit）＋FMI（5bit）|
| 故障情報の取得方法 | ツールから要求して取得 | ECUが定期ブロードキャスト（DM1） |
| 診断セッション概念 | あり（Default/Extended/Programming） | なし（常時診断情報を送信） |
| ランプ表示情報 | なし（UDS標準には含まない） | あり（DM1にMIL/AWL等のランプ状態を含む） |
| 標準規格 | ISO 14229 | SAE J1939-73 |

**注意**：建設機械ではJ1939 DMとUDSが**共存**するケースが多い。エンジン・変速機等の市販コンポーネントはJ1939 DMで故障通知し、車体コントローラにUDS対応を持たせてサービスツールからアクセスする、という構成が典型的。詳細は `diagnostics.md` を参照。

## 8. AUTOSARスタックへの影響

自動車向けAUTOSAR構成との差異。

| 比較項目 | 自動車CAN（乗用車）| J1939 |
|---|---|---|
| 基本COMスタック | AUTOSAR Com / CanIf / CanDrv | 同じ（共通） |
| TP処理 | CanTp（ISO 15765-2準拠） | **J1939Tp**（別モジュール） |
| ネットワーク管理 | CanNm（AUTOSAR NM） | **J1939Nm**（アドレスクレーム管理） |
| 診断 | Dcm + Dem（UDS対応） | Dem（DM1/DM2はDemが生成、J1939Dcmも存在） |
| ArXML設定 | 標準Com設定 | J1939固有パラメータ（J1939Tp・J1939Nm設定追加） |

**重要**：委託先へ要件定義する際、「J1939 TPとJ1939 Nmを有効にすること」を明示的に記載しないと、自動車向けのCanTp+CanNm構成のまま納品されるリスクがある。`requirements-spec.md` の通信要件セクションで明記すること。

## 9. 建設機械特有の状況：自動車CANとJ1939の混在

建設機械の現実のバス構成では、両者が混在することが多い。

```
建設機械の典型的なCAN構成

  ┌──────────────────────────────────────┐
  │  J1939バス（250kbps）                │
  │  エンジンECU ─ 変速機ECU ─ 車体コントローラ │
  │  （市販コンポーネント）               │
  └──────────────────────────────────────┘

  ┌──────────────────────────────────────┐
  │  社内開発CANバス（独自、500kbps等）  │
  │  油圧コントローラ ─ HMI ─ センサECU  │
  │  （独自プロトコルまたは簡易J1939）   │
  └──────────────────────────────────────┘

  ゲートウェイECU（2バス間を橋渡し）
```

| 信号の種類 | プロトコル | 根拠 |
|---|---|---|
| エンジン回転数・トルク | J1939 標準PGN（EEC1 等） | エンジンメーカがJ1939対応で出荷 |
| 変速機ギア位置・圧力 | J1939 標準PGN（ETC1 等） | 変速機メーカがJ1939対応で出荷 |
| バケット圧力・アーム角度 | J1939 Proprietary または独自 | J1939-75に定義があるが使われないことも多い |
| HMI表示・オペレータ入力 | 独自（J1939ベースとは限らない） | 社内開発で自由に決めているケースが多い |
| 人検知カメラ・GNSS | Ethernet（CAN系ではない） | 帯域不足のためEthernet化が進んでいる |

## 10. まとめ：自動車CANエンジニアが知っておくべきポイント

| 自動車CANの常識 | J1939での変化点 |
|---|---|
| IDはプロジェクト固有の割り当て | IDに構造（Priority/PGN/SA）が埋め込まれている |
| DBCがなければIDの意味は分からない | 標準PGNはSAE規格を見れば意味が分かる |
| ビットレートは500kbps/1Mbpsが普通 | 標準は250kbps（遅いが設計思想が異なる） |
| 診断はUDS/DCMでセッション管理 | 故障情報はDM1がECU側から定期ブロードキャスト |
| ISO 15765-2（UDS-TP）で8バイト超え対応 | J1939 TP（BAM/CMDT）で対応。フロー制御の有無が違う |
| ネットワーク管理はCANNMで省電力管理 | J1939NMはアドレスクレーム（SA競合解決）が主目的 |
| AUTOSARはCanTp+CanNmで構成 | J1939Tp+J1939Nmが追加で必要 |
| 同じエンジン回転数信号の定義は各社バラバラ | SPN 190がエンジン回転数と規格で固定されている |
